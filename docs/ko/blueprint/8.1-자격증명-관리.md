# 8.1 크리덴셜 관리

## 한 줄 정의

크리덴셜 관리는 iFay의 **열쇠 꾸러미**입니다—iFay가 당신을 대신하여 각종 서비스를 사용하는 데 필요한 "열쇠"를 안전하게 보관합니다. 하지만 이 열쇠는 원본이 아니라 안전한 사본입니다—사본을 잃어도 원본 열쇠는 여전히 안전합니다.

## 왜 필요한가

일상생활을 생각해 보세요: 집 열쇠, 차 열쇠, 사무실 출입 카드, 은행 카드 비밀번호, 각종 앱 로그인 계정…이 모든 것이 "크리덴셜"입니다—"당신이 무언가를 사용할 권리가 있음"을 증명하는 증거입니다.

이제 iFay가 당신을 대신하여 일하게 하고 싶습니다—전자상거래 플랫폼에 로그인하여 물건 사기, 회사 이메일로 메일 보내기, 스마트 홈 제어하기. iFay가 이런 일을 하려면 당신의 "열쇠"가 필요합니다. 하지만 집 열쇠 원본을 비서에게 주시겠습니까?

더 현명한 방법은: **사본 열쇠**를 비서에게 주는 것입니다. 사본으로 문을 열 수 있지만, 사본을 잃어도 즉시 폐기할 수 있고 원본 열쇠는 영향받지 않습니다. 이것이 크리덴셜 관리의 핵심 아이디어입니다—**사본 메커니즘**.

iFay의 크리덴셜 관리는 단순한 비밀번호 관리자가 아닙니다. 7가지 다른 유형의 크리덴셜을 관리하고, 각 크리덴셜에 대해 안전한 사본을 생성하며, 각 크리덴셜의 소유자(당신의 것인지 iFay 자체의 것인지)를 표시하고, 매번 사용의 감사 로그를 기록하며, 이상을 감지하면 자동으로 사본을 폐기합니다. iFay 보안 체계의 초석입니다.

## 아키텍처에서의 위치

```
iFay 4레이어 아키텍처
├── 소셜 레이어          ← 크리덴셜 관리는 여기
│   ├── FayID              ← 크리덴셜 유형 중 하나이자 iFay의 신원 식별
│   └── 👉 크리덴셜 관리    ← iFay의 "열쇠 꾸러미"
├── 인터랙션 레이어
│   └── 동작(Action)
│       ├── 시뮬레이션 조작  ← 시뮬레이션 조작 시 크리덴셜로 앱 로그인 필요
│       ├── 스킬 호출       ← 스킬 호출 시 크리덴셜 인증 필요
│       └── ...
├── 코그니션 레이어
│   └── 스킬(Skill)
│       └── 등록 스킬       ← 등록 스킬의 사전 인가가 크리덴셜에 의존
└── 에고 레이어
    └── Ego 모델
```

크리덴셜 관리는 **소셜 레이어**에 위치하며, FayID와 나란히 있습니다. 소셜 레이어가 정의하는 것은 iFay와 외부 세계의 "관계"입니다—당신이 누구인지(FayID), 무엇을 할 수 있는지(크리덴셜). 크리덴셜 관리는 아래로 인터랙션 레이어와 코그니션 레이어의 인증이 필요한 모든 조작에 크리덴셜 지원을 제공합니다.

## 작동 방식

크리덴셜 관리의 작동 방식은 네 가지 키워드로 요약됩니다: **분류, 복제, 표기, 수호**.

**1. 분류—7가지 크리덴셜 유형**

iFay가 관리하는 크리덴셜은 7가지 유형으로 나뉩니다(반복에 따라 더 추가될 수 있음):

![크리덴셜 관리](./illustration/credential-management.png)

- **신원 식별(FayID)**: iFay의 전역 고유 신원, 주민등록번호와 같음
- **계정/비밀번호**: 가장 흔한 크리덴셜 유형, 각종 앱과 서비스 로그인에 사용
- **인증서**: 디지털 인증서, 암호화 통신과 신원 확인에 사용
- **인가**: 제3자 인가(OAuth 등), iFay가 당신을 대표하여 특정 서비스에 접근하도록 허용
- **액세스 토큰**: 임시 접근 크리덴셜, 보통 유효 기간이 있음
- **스마트 컨트랙트**: 블록체인 상의 스마트 컨트랙트 크리덴셜, 탈중앙화 서비스에 사용
- **디지털 토큰(MeriToken)**: 사회적 기여도에 대응하는 디지털 토큰, iFay 생태계의 가치 교환에 사용

**2. 복제—사본 메커니즘**

크리덴셜을 iFay에 위임할 때, 크리덴셜 관리는 원본 크리덴셜을 직접 사용하지 않습니다. 원본 크리덴셜을 **대응하는 사본으로 교환**하며, iFay는 사본을 사용하여 로그인과 인증을 수행합니다.

**3. 표기—누구의 열쇠인가**

각 크리덴셜은 원래 소유자가 **휴먼 프라임(Human Prime)**인지 **iFay 자체**인지를 표시합니다.

휴먼 프라임(Human Prime) 권한과 iFay 권한 사이에는 포함 관계가 없습니다—iFay는 당신이 갖지 않은 권한을 가질 수 있고, 당신도 iFay에 위임하지 않은 권한을 가질 수 있습니다.

**4. 수호—보안 방어선**

- **감사 추적**: 크리덴셜의 매번 사용이 기록되며, 언제든 iFay가 당신의 크리덴셜로 무엇을 했는지 확인 가능
- **이상 감지**: 크리덴셜 사본의 이상 사용이 감지되면(예: 알 수 없는 위치에서 로그인) 시스템이 즉시 해당 사본을 폐기하고 알림
- **프라이버시 보호**: iFay가 프라이버시 데이터 조회를 인가받았을 때, **불리언 결과**(참 또는 거짓)만 피드백하며 실제 데이터를 노출하지 않음

## 시나리오 스토리

### 시나리오 1: 전자상거래 플랫폼 계정 위임—사본 메커니즘의 보안 보장

iFay에게 전자상거래 플랫폼에서 생활용품을 사게 하고 싶습니다. 전자상거래 플랫폼 계정을 iFay에 위임해야 합니다.

위임 과정에서 크리덴셜 관리가 하는 일:
1. 전자상거래 플랫폼 계정 비밀번호(원본 크리덴셜) 수신
2. 대응하는 **사본 크리덴셜** 생성—이 사본은 전자상거래 플랫폼 계정에 로그인할 수 있지만, 권한이 "생활용품 카테고리만, 건당 200원 이하"로 제한
3. 사본 크리덴셜을 안전하게 저장하고, 소유자를 "휴먼 프라임(Human Prime)"으로 표기
4. 원본 비밀번호는 iFay가 직접 보유하지 않음

한 달 후, 크리덴셜 관리가 이상을 감지합니다: 누군가 이 사본 크리덴셜로 5000원짜리 TV를 구매하려 합니다—인가 범위를 초과합니다. 크리덴셜 관리가 즉시 구매 요청을 거부하고, 사본 크리덴셜을 폐기하고, 알림을 보냅니다.

원본 비밀번호는 시종 안전합니다.

### 시나리오 2: 프라이버시 조회—참 또는 거짓만 반환

회사 HR로서 구직자가 제공한 학력 정보를 확인해야 합니다. iFay에게 조회를 맡깁니다.

iFay가 학력 확인 데이터베이스에 접근 인가를 받았습니다. 조회를 발행합니다: "학력 증서 번호 XXXXX가 유효한가?"

데이터베이스가 완전한 학력 정보를 반환했습니다—이름, 학교, 전공, 졸업 시기…하지만 크리덴셜 관리의 프라이버시 보호 메커니즘이 개입합니다. iFay는 결과 하나만 피드백합니다: **"예, 해당 학력 증서는 유효합니다."**

구직자의 다른 프라이버시 정보는 보이지 않으며, iFay도 이 정보를 저장하지 않습니다.

## 개발자를 위해

크리덴셜 관리는 **1단계(인간 조작 시뮬레이션)**의 핵심 모듈입니다.

- **요구사항 번호**: 요구사항 3(크리덴셜 관리)
- **인터페이스 규격**: `CredentialManager` 인터페이스, `delegate()`, `authenticate()`, `revoke()`, `queryPrivacy()`, `getAuditLog()` 5개 핵심 메서드 포함
- **7가지 크리덴셜 유형**: `fay_id`, `account_password`, `certificate`, `authorization`, `access_token`, `smart_contract`, `meri_token`
- **적합성 테스트**: iFACTS L1이 7가지 크리덴셜 유형 관리 능력과 사본 메커니즘을 검증, L2가 등록 스킬 및 스킬 호출과의 인증 인터페이스를 검증, L4가 프라이버시 조회의 불리언 반환 제약과 이상 감지 행동을 검증
