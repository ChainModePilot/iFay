# 8.1 Управление учётными данными

## Определение в одну строку

Управление учётными данными — это **связка ключей** iFay — оно безопасно хранит «ключи», необходимые iFay для использования различных сервисов от вашего имени. Но эти ключи — не ваши оригинальные ключи, а безопасные копии. Даже если копия потеряна, ваши оригинальные ключи остаются в безопасности.

## Зачем это нужно

Подумайте о вашей повседневной жизни: у вас есть ключи от дома, ключи от машины, пропуск в офис, PIN-коды банковских карт, аккаунты для входа в различные приложения… Всё это — «учётные данные» — доказательство того, что «вы имеете право использовать что-то».

Теперь вы хотите, чтобы iFay делал что-то от вашего имени — входил на платформу электронной коммерции для покупок, использовал вашу корпоративную почту для отправки сообщений, управлял вашим умным домом. Для этого iFay нужны ваши «ключи». Но отдали бы вы оригинальный ключ от дома помощнику?

Более умный подход: сделать **копию ключа** для помощника. Копия может открыть дверь, но если копия потеряна, вы можете немедленно её аннулировать, а ваш оригинальный ключ останется нетронутым. Это основная идея Управления учётными данными — **механизм копирования**.

Управление учётными данными iFay — это больше, чем менеджер паролей. Оно управляет семью различными типами учётных данных, создаёт безопасные копии для каждого, отмечает владельца каждого учётного данного (ваш или собственный iFay), записывает аудиторские журналы каждого использования и автоматически отзывает копии при обнаружении аномалий. Это краеугольный камень системы безопасности iFay.

## Его место в архитектуре

```
Четырёхслойная архитектура iFay
├── Социальный слой          ← Управление учётными данными здесь
│   ├── FayID              ← Один из типов учётных данных, также идентичность iFay
│   └── 👉 Управление учётными данными  ← «Связка ключей» iFay
├── Слой взаимодействия
│   └── Действие
│       ├── Имитация операций  ← Нужны учётные данные для входа в приложения
│       ├── Вызов навыков      ← Нужны учётные данные для аутентификации
│       └── ...
├── Когнитивный слой
│   └── Навыки
│       └── Зарегистрированные навыки  ← Предварительная авторизация зависит от учётных данных
└── Слой Ego
    └── Модель Ego
```

Управление учётными данными находится в **Социальном слое**, рядом с FayID. Социальный слой определяет «отношения» iFay с внешним миром — кто вы (FayID) и что вам разрешено делать (учётные данные). Управление учётными данными предоставляет поддержку учётных данных вниз для всех операций, требующих аутентификации, в Слое взаимодействия и Когнитивном слое.

## Как это работает

Рабочий процесс Управления учётными данными можно резюмировать четырьмя ключевыми словами: **Классифицировать, Копировать, Пометить, Защитить**.

**1. Классифицировать — Семь типов учётных данных**

iFay управляет семью типами учётных данных (с итерациями могут быть добавлены новые):

![Управление учётными данными](./illustration/credential-management.png)

- **Идентификатор личности (FayID)**: Глобально уникальная идентичность iFay, как ваш номер паспорта
- **Аккаунт / Пароль**: Наиболее распространённый тип учётных данных для входа в различные приложения и сервисы
- **Сертификат**: Цифровые сертификаты для зашифрованной коммуникации и верификации личности
- **Авторизация**: Авторизация третьей стороны (например, OAuth), позволяющая iFay получать доступ к определённым сервисам от вашего имени
- **Токен доступа**: Временные учётные данные доступа, обычно с периодом истечения
- **Смарт-контракт**: Учётные данные блокчейн-смарт-контрактов для децентрализованных сервисов
- **Цифровой токен (MeriToken)**: Цифровые токены, соответствующие социальному вкладу, для обмена ценностью в экосистеме iFay

**2. Копировать — Механизм копирования**

Когда вы делегируете учётные данные iFay, Управление учётными данными не использует напрямую ваши оригинальные учётные данные. Оно **обменивает оригинал на соответствующую копию**, и iFay использует копию для входа и аутентификации.

Преимущества этого дизайна:
- Если копия скомпрометирована, вы можете немедленно её отозвать, а оригинал останется нетронутым
- Копии могут иметь более строгие области разрешений (например, только чтение, без модификаций)
- Использование копий может быть полностью отслежено и проаудировано

**3. Пометить — Чей ключ**

Каждый тип учётных данных указывает, является ли его первоначальным владельцем **Human Prime** (человеческий оригинал) или **сам iFay**.

Это важно, потому что iFay не только использует ваши учётные данные — он также может владеть собственными. Между разрешениями Human Prime и разрешениями iFay нет отношения вложенности — iFay может иметь разрешения, которых нет у вас, а вы можете иметь разрешения, не делегированные iFay.

**4. Защитить — Линия обороны безопасности**

- **Аудиторский след**: Каждое использование учётных данных регистрируется
- **Обнаружение аномалий**: При обнаружении аномального использования копии система немедленно отзывает копию и уведомляет вас
- **Защита конфиденциальности**: Когда iFay авторизован для запроса приватных данных, он возвращает только **булевые результаты** (истина или ложь) без раскрытия фактических данных

## Связи с другими модулями

| Связанный модуль | Связь | Аналогия с телом |
|-----------------|-------|------------------|
| **FayID** | FayID — один из семи типов учётных данных и основная идентичность iFay | Паспорт — самый базовый документ |
| **Зарегистрированные навыки** | Предварительная авторизация зарегистрированных навыков зависит от учётных данных аутентификации | Нужно показать документы, чтобы начать работу |
| **Имитация операций** | Имитация операций нуждается в учётных данных при входе в приложения | Нужны ключи, чтобы открыть двери |
| **Вызов навыков** | Вызов навыков нуждается в учётных данных для аутентификации при выполнении задач | Нужно принести документы при оформлении дел |
| **Профиль iFay** | Информация об учётных данных записывается в измерении разрешений Профиля | Список документов — часть удостоверения личности |

## Сценарии

### Сценарий 1: Делегирование аккаунта платформы электронной коммерции

Вы хотите, чтобы iFay покупал товары первой необходимости на платформе электронной коммерции от вашего имени. При делегировании Управление учётными данными:
1. Получает ваш пароль аккаунта (оригинальное учётное данное)
2. Генерирует соответствующую **копию** — эта копия может входить в ваш аккаунт, но разрешения ограничены «только категория товаров первой необходимости, одна покупка не более 200 юаней»
3. Безопасно хранит копию, помеченную владельцем «Human Prime»
4. Ваш оригинальный пароль никогда напрямую не хранится у iFay

Месяц спустя обнаруживается аномалия: кто-то попытался использовать эту копию для покупки телевизора за 5000 юаней — превышение авторизованной области. Управление учётными данными немедленно отклоняет запрос, отзывает копию и отправляет вам уведомление. Ваш оригинальный пароль остаётся в безопасности.

### Сценарий 2: Запрос конфиденциальности — возвращает только истину или ложь

Вы — HR-менеджер и нужно проверить образовательные данные соискателя. iFay авторизован для доступа к базе данных верификации образования. Он инициирует запрос: «Действителен ли сертификат об образовании номер XXXXX?»

База данных возвращает полную образовательную информацию, но механизм защиты конфиденциальности вмешивается. iFay возвращает вам только один результат: **«Да, сертификат об образовании действителен.»**

Философия этого дизайна: **iFay имеет право запрашивать, но не имеет права раскрывать**.

## Для разработчиков

Управление учётными данными относится к **Фазе 1 (Имитация действий человека)** как основной модуль.

- **ID требования**: Требование 3 (Управление учётными данными)
- **Спецификация интерфейса**: Интерфейс `CredentialManager` с пятью основными методами: `delegate()`, `authenticate()`, `revoke()`, `queryPrivacy()`, `getAuditLog()`
- **Семь типов учётных данных**: `fay_id`, `account_password`, `certificate`, `authorization`, `access_token`, `smart_contract`, `meri_token`
- **Владелец учётных данных**: `host` (Human Prime) или `ifay`
- **Статус копии**: `active`, `revoked`, `expired`
- **Тестирование соответствия**: iFACTS L1 верифицирует управление семью типами и механизм копирования; L2 верифицирует интерфейсы аутентификации; L4 верифицирует ограничение булевого возврата для запросов конфиденциальности
