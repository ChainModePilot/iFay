# 12.2 Habilidades registradas

✅ "Habilidades registradas" se refiere a una habilidad que iFay ya ha dominado u obtenido la capacidad de usar.

❌ No significa registrar o crear una cuenta en ningún sistema externo.

## Definición en una línea

Las Habilidades registradas son la **certificación de habilidades** de iFay — así como los humanos necesitan una licencia de conducir para conducir o una licencia médica para ejercer medicina, iFay necesita "registrar" una habilidad antes de poder usarla. El registro no es solo un simple registro — es preautorización, asegurando que no se necesite autenticación adicional en el momento de la ejecución.

## Por qué es necesario

El registro es un prerrequisito para cualquier acción de iFay. Cuando una habilidad se registra en iFay, significa que iFay puede invocarla en cualquier momento. El proceso de registro es como obtener una certificación:

- **Confirmar capacidad**: ¿Qué tipo es esta habilidad? ¿Qué puede hacer?
- **Completar preautorización**: Obtener permiso de uso por adelantado, para que la ejecución real no necesite pasar por el proceso de autenticación de nuevo
- **Registrar en archivo**: El Perfil de iFay registra todas las habilidades registradas

¿Por qué es tan importante la "preautorización"? Imagina que eres un médico de emergencias y llega un paciente — no puedes decir "espera, déjame ir a buscar mi licencia primero." La preautorización asegura que iFay pueda **actuar inmediatamente** cuando necesita ejecutar una tarea, sin retrasos por procesos de autenticación.

iFay soporta seis tipos de habilidades: **API, Workflow, Bot, Agent, APP, Microservicio**. Independientemente del tipo, el registro es el prerrequisito para el uso — sin registro, sin derechos de ejecución.

## Cómo funciona

La operación de las Habilidades registradas se puede resumir en cuatro palabras clave: **Registrar, Preautorizar, Cachear, Refrescar**.

**1. Registrar — "Incorporación" de habilidad**

Cuando iFay necesita usar una nueva habilidad, el registro debe completarse primero. El proceso de registro registra información clave de la habilidad: nombre y tipo, endpoint de acceso, método de autenticación requerido y descripción de capacidad.

**2. Preautorizar — Obtener el "pase" por adelantado**

El valor central del registro radica en la preautorización. Durante el registro, iFay completa todos los pasos de autenticación por adelantado — obteniendo tokens de acceso, estableciendo conexiones seguras, verificando permisos. De esta manera, cuando una tarea realmente necesita ejecutarse, iFay puede llamarla directamente con cero retraso.

**3. Cachear — Funciona sin conexión también**

iFay no siempre está en línea. Cuando iFay está sin conexión, el módulo de Habilidades registradas cachea las acciones pendientes. Una vez que la red se recupera, estas acciones cacheadas se ejecutan asincrónicamente.

**4. Refrescar — Mantener la autorización válida**

La preautorización no es permanente. Los tokens de acceso expiran, los permisos pueden cambiar. El módulo de Habilidades registradas refresca periódicamente el estado de autorización, asegurando que los permisos de cada habilidad sean válidos cuando se llaman.

## Historias de escenarios

### Escenario 1: Registrar una API de traducción — respuesta instantánea por preautorización

Frecuentemente necesitas comunicarte con clientes en el extranjero, así que haces que iFay registre una habilidad de API de traducción profesional.

Durante el registro, iFay completa los siguientes pasos:
- Registra el tipo de la API de traducción (habilidad tipo API), dirección del endpoint y descripción de capacidad
- Usa tus credenciales delegadas para completar la autenticación de API y obtener un token de acceso
- Almacena de forma segura el token y marca el estado de preautorización como "completado"

Una semana después, recibes un correo en japonés. Le dices a iFay: "Ayúdame a traducir este correo."

Porque la habilidad de traducción ya fue registrada y preautorizada, iFay **inmediatamente** llama a la API de traducción, presentándote el resultado de la traducción en dos segundos. Sin ventana emergente pidiendo tu contraseña, sin rueda girando esperando autenticación — todo fue preparado durante el registro.

### Escenario 2: Caché sin conexión en un avión

Estás en un vuelo a Nueva York, teléfono en modo avión. Durante el vuelo, piensas en varias cosas que necesitas que iFay maneje:

- "Envía un correo al Director Zhang confirmando la hora de la reunión del próximo miércoles"
- "Reserva el restaurante japonés mejor calificado cerca de mi hotel en Nueva York"
- "Revisa el clima de mañana en Nueva York"

iFay recibe estas tres solicitudes. Las tres habilidades ya están registradas, pero no hay red ahora mismo.

El módulo de Habilidades registradas cachea estas tres acciones pendientes y te dice: "Entendido. Estas tres cosas requieren acceso a internet para completarse. Las manejaré inmediatamente después de aterrizar."

El avión aterriza, enciendes tu teléfono y la red se recupera. El módulo de Habilidades registradas detecta la conexión restaurada y ejecuta inmediatamente las tres acciones cacheadas asincrónicamente: correo enviado, restaurante reservado, información del clima enviada a ti.

## Para desarrolladores

Las Habilidades registradas pertenecen a la **Fase 2 (Toma de control directo del cliente)** como módulo central.

- **ID de requisito**: Requisito 9 (Gestión de Habilidades registradas)
- **Especificación de interfaz**: Interfaz `RegisteredSkillManager` con cinco métodos centrales: `register()`, `query()`, `refreshAuthorization()`, `cacheOfflineAction()` y `flushCachedActions()`
- **Seis tipos de habilidades**: `api`, `workflow`, `bot`, `agent`, `app`, `microservice`
- **Estados de preautorización**: `pre_authorized`, `pending`, `expired`
- **Módulos asociados**: Invocación de habilidades (`InvokeSkillService`) consume habilidades registradas, Gestión de Credenciales (`CredentialManager`) proporciona soporte de autenticación, Habilidades internas (`InternalSkill`) proporciona capacidades complementarias
- **Pruebas de conformidad**: iFACTS L1 valida registro y preautorización para los seis tipos de habilidades, L2 valida integración de interfaz con Invocación de habilidades y Gestión de Credenciales, L3 valida el flujo completo de caché sin conexión y ejecución de recuperación
