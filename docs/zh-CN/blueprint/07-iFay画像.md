# 7. iFay 画像

iFay Profile 是 iFay 的完整"身份证"——一个语义可解释的六维属性表。它不仅让人类能够识别一个 iFay，也让系统能够解析它，更让 Fay 之间能够相互认识。

人类原型画像（对齐意识 / Aligned Consciousness）是 iFay Profile 的子集和输入源。人类原型画像描述的是"人类原型是什么样的人"，而 iFay Profile 描述的是"这个 iFay 是什么样的存在"——它包含了人类原型的人格特征，但也包含了 iFay 自身的技能、硬件、权限等独立属性。

人类原型画像由[对齐意识](./11-认知层-思维#113-对齐意识)模块维护。技能维度由[注册技能](./12-认知层-技能#122-注册技能)管理，硬件维度由[设备驱动中枢](./12-认知层-技能#121-设备驱动中枢)管理，权限维度由[凭证管理](./08-社交层#82-社会权限)管理。

---

## iFay 身份

iFay 身份由 FayID 和元信息组成。FayID 是全局唯一、人类可读、机器可读的持久标识符，是 iFay 在所有交互场景中被识别和追踪的基础。

iFay 身份支持平滑的身份迁移——当 iFay 从个人助手演变为自主社会角色时，FayID 不需要更换，身份自然延续。

---

## Ego 模型

Profile 中的 Ego 维度记录当前活跃的 Ego 版本引用。

Ego 模型是可插拔、可切换的边缘小模型，加载到任何终端设备上即可让该设备拥有人类原型的个性。iFay 支持拥有多个 Ego 版本，以适应人类原型的多面人格——但任一时刻只有一个版本处于活跃状态，防止"人格分裂"。

切换方式有两种：人类原型手动触发，或 iFay 基于上下文自动判断。切换完成后，Profile 中的活跃 Ego 版本引用会同步更新。

### 场景

> 张磊是一名产品经理，他为自己的 iFay 配置了两个 Ego 版本："专业模式"和"休闲模式"。
>
> 工作日上午，iFay 自动切换到"专业模式"——回复邮件措辞严谨，安排会议时优先考虑效率，处理需求文档时注重逻辑完整性。
>
> 周末晚上，iFay 检测到张磊在浏览美食推荐，自动切换到"休闲模式"——推荐餐厅时更注重氛围和口味偏好，语气也变得轻松随意。
>
> 两个版本，同一个张磊。

## Faying 思维

Faying 思维是 iFay 的认知资产，包含四种子类型：

| 子类型 | 说明 | 示例 |
|--------|------|------|
| **内容（Content）** | 结构化或非结构化的内容资产 | 人类原型的照片集、撰写的文章、创作的音乐、录制的视频 |
| **数据（Data）** | 个人数据 | 健康监测数据、消费记录、应用使用习惯、位置历史 |
| **知识库（Knowledge Base）** | 组织化的知识体系 | 专业领域笔记、学习资料库、工作经验总结、行业研究报告 |
| **信息流（Info Feed）** | 实时信息订阅 | 关注的新闻源、行业动态推送、社交媒体更新、股票行情 |

思维维度的数据来源于个人数据堆的初始化和持续积累，是 iFay 理解世界和做出判断的基础。

## Faying 技能

Faying 技能是 iFay 的行动能力，包含六种技能类型：

| 类型 | 说明 |
|------|------|
| **API** | 可直接调用的编程接口，如天气查询 API、翻译 API |
| **工作流（Workflow）** | 多步骤的自动化流程，如"每周一生成周报并发送给团队" |
| **Bot** | 特定场景的对话机器人，如客服 Bot、预约 Bot |
| **Agent** | 具备自主决策能力的智能代理，如投资分析 Agent |
| **APP** | 完整的应用程序，如日历管理 APP、健康追踪 APP |
| **微服务（Microservice）** | 独立部署的服务单元，如图像识别服务、语音转文字服务 |

技能注册是 iFay 执行任何动作的前提条件。已注册的技能会完成预授权，确保执行时无需额外认证。

## Faying 硬件

硬件是 iFay 的"肢体"。iFay 是人格和认知实体，硬件是它与物理世界交互的载体。

### 三种硬件子类型

| 子类型 | 说明 | 示例 |
|--------|------|------|
| **设备（Device）** | 终端硬件设备 | 手机、电脑、智能手表、无人机、智能家居设备 |
| **存储（Storage）** | 数据存储资源 | 云存储、本地硬盘、NAS、向量数据库 |
| **算力（Computing）** | 计算资源 | GPU、云计算节点、边缘计算设备 |

### 硬件发现机制

当 iFay 迁移到新环境时，终端设备负责扫描当前环境中可 Faying 的硬件——类似蓝牙或 WiFi 扫描周围设备的方式。iFay 本身不直接执行硬件扫描，而是根据扫描结果进行能力匹配。

这里有一个关键区分：

- **iFay 能操控哪些硬件**，取决于它的认知能力（注册技能、内部技能）
- **iFay 能连接到哪些硬件**，取决于当前环境中实际可用的设备

### 场景

> 想象一个会打羽毛球的人走进一间体育馆。
>
> "会打羽毛球"是他的**技能**——这是他自身的能力，无论走到哪里都带着。但他能不能打球，还取决于体育馆里有没有**球拍**——这是环境提供的硬件。
>
> iFay 也是如此。它可能拥有控制无人机的技能（注册了飞行控制 API），但如果当前环境中没有可连接的无人机，这个技能就无法施展。反过来，即使环境中有一台 3D 打印机，如果 iFay 没有注册相关的操控技能，它也只能"看到"这台设备却无法使用。

## Faying 权限

权限是 iFay 的"关系"——它定义了 iFay 与外部世界的信任边界。

### 三种权限生命周期

| 生命周期 | 说明 | 示例 |
|----------|------|------|
| **与生俱来（Inherent）** | 创建时即拥有的权限 | 访问人类原型日历、读取人类原型偏好设置 |
| **持久（Persistent）** | 长期有效的权限 | 人类原型邮箱的读写权限、企业内网访问权 |
| **临时（Ephemeral）** | 需要时申请，用后释放 | 一次性访问医疗记录、临时获取某个 API 的调用权 |

### 可扩展的鉴权方式

iFay 的鉴权方式是可扩展的，支持 SSO、OAuth、Fingerprint 等方法。这里的 Fingerprint 是一个统称，涵盖一切可识别身份的标识——面部识别、声纹、设备指纹等。不同的指纹类型具有不同的确信力度：面部识别的确信力度可能高于设备指纹，而多因素组合的确信力度又高于单一因素。

### 场景

> 王芳的 iFay 拥有三种不同生命周期的权限：
>
> **与生俱来**：iFay 创建时就自动获得了访问王芳日历的权限——这是每个 iFay 的基本能力，不需要额外申请。
>
> **持久权限**：王芳授权 iFay 长期管理她的工作邮箱。iFay 可以阅读邮件、起草回复、管理标签，这个权限一直有效，直到王芳主动撤销。
>
> **临时权限**：王芳需要查看去年的体检报告。她让 iFay 去医院系统调取记录。iFay 临时申请了一次性的医疗数据访问权限，获取报告后权限自动释放——下次再需要时，需要重新申请。

## Profile 可见性

Profile 的可见性是一个应用层关注点，由具体的应用场景和交互方式决定。

同一个 iFay Profile，在不同场景下展示不同的维度：在 Fay 间协作时，对方可能只需要看到技能和权限维度；在人类原型管理界面中，所有六个维度都完整可见；在公开社交场景中，可能只展示身份和部分技能信息。

Profile 本身不定义"谁能看到什么"——这个决策交给使用 Profile 的应用来做。
