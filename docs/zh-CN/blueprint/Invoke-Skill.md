# 技能调用

## 一句话定义

技能调用是 iFay 的**嘴巴和电话**——如果模拟操作是 iFay 用手操作界面，技能调用就是 iFay 直接"打电话"给服务——不需要操作界面，直接调用 API 或触发任务。

## 为什么需要它

想象你要翻译一份文件。你有两种方式：

**方式一**：打开浏览器，进入翻译网站，把文字粘贴进去，等待翻译，复制结果。这就是"模拟操作"的方式——通过界面一步步操作。

**方式二**：直接拨打翻译公司的电话，说"帮我翻译这段话"，对方翻译好直接告诉你结果。不需要打开任何网站，不需要操作任何界面。这就是"技能调用"的方式——**直接打电话给服务**。

为什么需要第二种方式？因为很多时候，操作界面是多余的中间环节。你真正想要的不是"打开翻译网站"，而是"把这段话翻译成英文"。技能调用让 iFay 跳过界面，直接完成你的目标。

这就像人类社会中的分工：你不需要亲自去面包店揉面团（操作界面），你只需要打个电话说"送一个面包过来"（调用技能）。技能调用让 iFay 能够高效地利用已经注册好的各种服务——API、工作流、Bot、Agent、App、微服务——就像一个善于打电话办事的助手。

## 它在架构中的位置

```
iFay 四层架构
├── 社交层
├── 交互层          ← 技能调用在这里
│   ├── 感知（Sense）
│   │   ├── 第一人称追踪器   ← 眼睛
│   │   ├── 传感器           ← 神经系统
│   │   └── 自我感知         ← 内心
│   └── 动作（Action）
│       ├── 模拟操作         ← 双手：操作界面
│       ├── 👉 技能调用      ← 嘴巴/电话：直接调用服务
│       └── 自驱行为         ← 自主意志
├── 认知层
│   └── 技能
│       ├── 注册技能         ← 技能调用的"电话簿"
│       └── ...
└── 自我层
```

技能调用位于**交互层的动作子系统**中，和模拟操作、自驱行为并列。它与认知层的**注册技能**模块关系最为紧密——注册技能是 iFay 的"电话簿"，记录了所有可以调用的服务；技能调用则是"拨打电话"的动作本身。

一个关键规则：**只有在电话簿里登记过的号码，才能拨打**。也就是说，技能必须先注册，才能被调用。这确保了 iFay 不会随意调用未经授权的服务。

## 它是怎么工作的

技能调用的工作方式可以用三个关键词概括：**匹配、调用、记录**。

**1. 匹配——理解你想做什么，找到对应的技能**

当你对 iFay 说"帮我把这段话翻译成英文"时，iFay 需要理解你的意图（翻译），然后在已注册的技能中找到匹配的服务（比如一个翻译 API）。

这个过程叫做**意图匹配**。iFay 不是简单地做关键词搜索，而是理解你的真实需求，然后找到最合适的技能来完成。

**2. 调用——直接执行**

找到匹配的技能后，iFay 直接调用它——就像拨通电话一样。调用的方式取决于技能的类型：
- 如果是 **API**，就发送一个 API 请求
- 如果是**工作流**，就触发一个多步骤的自动化流程
- 如果是 **Bot 或 Agent**，就向它发送指令
- 如果是 **App**，就启动对应的应用功能
- 如果是**微服务**，就调用对应的服务接口

**3. 记录——每次调用都留痕**

每次技能调用的过程、结果和评估信息都会被记录下来。这不仅是为了审计和追溯，更重要的是为 **GMChain（全球贡献链）** 提供贡献追踪数据——每次调用一个技能，提供这个技能的人或组织就获得了相应的贡献记录。

**如果技能没注册或过期了怎么办？**

iFay 会拒绝执行，并告诉你原因："这个翻译服务的授权已经过期了，需要重新授权才能使用。"它不会偷偷调用未授权的服务，也不会在授权过期后继续使用。

## 与其他模块的关系

| 相关模块 | 关系 | 人体类比 |
|---------|------|---------|
| **注册技能** | 技能调用的前提——只有注册过的技能才能被调用 | 电话簿里有号码，才能打电话 |
| **模拟操作** | 同属动作子系统，互补关系：模拟操作走界面，技能调用走 API | 用手操作 vs 打电话办事 |
| **自驱行为** | 自驱行为可以自动触发技能调用 | 自主意志决定"该打这个电话了" |
| **Ego 模型** | Ego 模型约束技能调用的行为（比如不调用与宿主价值观冲突的服务） | 性格决定"哪些电话该打，哪些不该打" |
| **GMChain** | 每次技能调用都记录贡献，为价值分配提供依据 | 每次打电话办事都记账 |

## 场景故事

### 场景一：直接调用翻译 API 翻译文档

你收到一份法语的商业合同，需要翻译成中文。你对 iFay 说："帮我把这份合同翻译成中文。"

如果用模拟操作，iFay 需要：打开翻译网站 → 粘贴文字 → 选择语言 → 等待翻译 → 复制结果。一份 20 页的合同，可能要来回粘贴很多次。

但通过技能调用，iFay 直接"打电话"给翻译服务：
1. iFay 理解你的意图：翻译法语合同为中文
2. 在注册技能中找到一个专业文档翻译 API（已预授权）
3. 直接将整份合同发送给翻译 API
4. 几秒钟后收到完整的中文翻译结果
5. 将翻译结果整理好呈现给你
6. 这次调用被记录在案——翻译服务的提供者获得了一笔贡献记录

整个过程不需要打开任何网页，不需要操作任何界面。就像你打了一个电话，对方帮你搞定了一切。

### 场景二：调用公司会议室预订工作流

周一早上，你对 iFay 说："帮我订一个明天下午两点的会议室，要能坐 8 个人的，有投影仪。"

iFay 开始工作：
1. 理解你的意图：预订会议室，条件是明天下午两点、8 人容量、有投影仪
2. 在注册技能中找到公司的"会议室预订工作流"（这是一个多步骤的自动化流程）
3. 直接触发这个工作流，传入参数：日期、时间、人数、设备需求
4. 工作流自动执行：查询可用会议室 → 筛选符合条件的 → 预订 → 发送日历邀请
5. iFay 告诉你："已经帮你订好了 B 栋 305 会议室，明天下午两点到三点，8 人座，有投影仪。日历邀请已经发到你邮箱了。"

在这个场景中，iFay 没有打开公司的会议室预订系统网页，没有一个个点击筛选条件——它直接调用了公司内部的预订工作流，就像你打了一个电话给行政部门，说了你的需求，对方帮你搞定了一切。

## 对开发者

技能调用属于 **阶段 2（直接接管客户端）** 的核心模块。

- **需求编号**：需求 10（技能调用）
- **接口规范**：`InvokeSkillService` 接口，包含 `invokeByIntent()`（意图匹配调用）、`invokeDirect()`（直接调用）和 `getInvocationLog()`（调用日志）三个核心方法
- **前置依赖**：注册技能模块（`RegisteredSkillManager`）——技能必须先注册才能调用
- **支持的技能类型**：API、Workflow、Bot、Agent、APP、Microservice（六种）
- **关键行为**：未注册或预授权过期的技能将被拒绝调用；每次调用记录过程、结果和评估信息，支持 GMChain 贡献追踪
- **合规性测试**：iFACTS L1 验证意图匹配和技能调用能力，L2 验证与注册技能模块的接口对接，L3 验证"意图表达→技能调用→贡献记录"的完整链路
