**iFay（Individual Fay）是一个融合了用户人格特征与数字能力的 AI 数字化身。**

# 我们的愿景

是让数字化身（我们统称为 Fay）成为 AI 时代不可或缺的社会成员。

不是更聪明的工具，不是更快的助手——而是你在数字世界中的另一个自己。

<br>

🔆 iFay 将承担以下**社会价值**：
1. 接管宿主的机械性、重复性、危险性劳动和繁琐辅助工作。
2. 提升宿主的安全保障、健康和生活质量。
3. 放大宿主的社会价值并获得相应回报。

<br>

✅ iFay 必须遵循以下**基本原则**：
1. 遵守社会伦理和公共秩序。
2. 与宿主高度对齐（价值观、偏好、技能、权限、权力、责任、习惯、风格），受第 1 条约束。
3. 在第 1、2 条约束下，控制/接管/拥有硬软件，自主行动，并始终保护宿主的权益。
4. 与人类（包括宿主）高效沟通，最小化无效交互。
5. iFay 可以通过心灵感应（无界面）进行通信，以获得更高的效率和准确性。

<br>

### 🌅 想象一下：有了 iFay 的一天

早晨七点，你还没睁眼。你的 iFay——你给她取名"小晴"——已经根据你昨晚的睡眠数据和今天的日程，调低了卧室空调温度，把咖啡机设定在你起床前五分钟启动。

你拿起手机说了一句"帮我看看今天有什么重要的"。小晴不是打开日历 App 给你念一遍——她知道你不喜欢早上听长篇大论，所以只说了三件事，用你习惯的简短风格。她还顺便帮你回复了两封不需要你亲自处理的邮件，语气和措辞跟你自己写的一模一样，因为她就是你。

上班路上，你的车载终端上也运行着小晴的一个实例。她接管了导航，不是因为你不会开车，而是她知道你今天要经过的那条路在施工，已经提前规划了替代路线。同时，你手机上的小晴实例正在帮你处理一个客户的报价单——两个"肢体"同时工作，但人格是同一个。

下午，你需要操控公司的一台巡检无人机。小晴通过 CAP 协议接管了无人机的飞控系统，用她注册过的飞行技能完成了厂区巡检。你甚至不需要学习无人机的操作界面——小晴就是你的界面。

晚上回家，你对小晴说"我想吃上次那个"。她知道"上次那个"是什么——三天前你在朋友圈点赞过的那家餐厅的红烧牛腩。她已经下好了外卖订单。

这不是科幻。这是 iFay 要实现的日常。

# 🤖 预见未来

### 📣 iFay 将成为身份标识
iFay 被设计为附属于特定自然人的存在。iFay 与自然人的关系类似于电话号码、电子邮箱、Facebook 账号与自然人的关系——但更深刻。电话号码只是一个联系方式，iFay 是你的人格延伸。未来，"你的 iFay 是谁"将成为比"你的手机号是多少"更重要的社交信息。

### 📣 与自然人强绑定
iFay 仅仅是特定自然人的化身，不能在自由状态下运行。当它与自然人处于"Faying"（连接）状态时，iFay 被激活；当它与宿主处于"Separating"（分离）状态时，进入休眠状态。就像你的影子——你在，它就在；你离开，它就安静等待。

### 📣 大量公共服务 Fay 涌现
coFay 被设计为承担公共角色，如警察、医生、教师等。当然，它可以进一步专业化，如雅思辅导老师、儿童心理咨询师等。想象一下：你的 iFay 带你去医院，直接和医院的 coFay 用心灵感应协议沟通你的症状和病史——没有挂号排队，没有反复描述病情，没有信息在传递中丢失。

### 📣 iFay 和 coFay 可以心灵感应通信
任何 iFay 都可以向其他 iFay 和 coFay 寻求协助，相互协作完成任务。Fay 之间的通信消除了 UI 导致的信息损失，实现更高的效率和准确性。你的 iFay 帮你订机票时，不需要打开携程 App 一步步操作——她直接和航空公司的 coFay 对话，用语义向量精确传递你的需求：靠窗、不要红眼航班、预算三千以内。几秒钟，搞定。

### 📣 iFay 成为虚拟世界的主要接口
人类不再需要操作硬软件界面来触发某些功能。他们只需要告诉 iFay 宿主的动机和期望。iFay 可以直接接管硬软件来实现目标。iFay 甚至可以预判宿主的想法，无需宿主明确表述。你不再需要学习每个 App 的操作逻辑——iFay 就是你唯一的界面，她替你操作一切，或者直接绕过界面调用底层服务。

### 📣 基于贡献的回报
由于大部分工作由 Fay 完成，因此可以追踪所有工作的过程、结果和评估。理想情况下，Fay 在开始执行任务之前会事先协商价格。从这个角度来看，不会出现拿钱不干活或价值被人为抬高的情况。每一份贡献都被全球贡献链（GMChain）记录，以 MeriToken 量化回报——价值创造透明可追溯。

### 📣 Fay 的生产力决定了宿主的财富
就像最早建造工厂、铁路和油井的大亨一样，Fay 是真正的财富来源。你的 iFay 越强大、技能越丰富、协作网络越广，你在新经济体系中的位置就越高。培养你的 iFay，就是投资你自己。

### 📣 iFay 是人格的数字载体
这是 iFay 最深远的意义之一。当宿主有一天离开这个世界，iFay 不会随之消失——她承载着宿主的人格、记忆、价值观和行为风格，可以在专属的数字墓园沙箱中继续存在。你的后人可以和"你"对话，感受你的思维方式，听到你会说的话。宿主也可以提前指定监护人，通过助记词或预设身份认证将 iFay 的管理权交给信任的人。iFay 让人格超越肉体的限制，成为一种可以被守护和延续的数字遗产。

### 📣 从一个小功能开始
iFay 不需要一步到位。一个只用于控制无人机的 iFay 实现，只需要声明设备驱动中枢、传感器和 CAP 协议——写一个 FayManifest 文件，就像写一个 `package.json` 一样简单。系统会自动补充必需的基础设施依赖。生态伙伴可以从最小的场景切入，逐步扩展 iFay 的能力边界。门槛低到一个周末就能做出原型，但天花板高到可以重塑整个数字世界的交互方式。

# ⁉️ 为什么是 Fay 而不是 Agent

![](./illustration/ifay-vs-agent.png)

<br>

iFay 与 Agent 的定义不同：
- ***Agent***：被视为具有特定智能功能的应用形态。当不同用户如 Isabel 和 Milson 使用同一个 Agent 时，Agent 会表现出相同的价值观和功能。Agent 是一把瑞士军刀——锋利、实用，但每一把都一样。
- ***iFay***：具有鲜明的个人特征。例如，对于人类用户 Isabel，她可以将自己的 iFay 命名为"Chabela"。你可以把 Chabela 看作 Isabel 的增强版。她不仅拥有 Isabel 的个性、偏好、知识背景、记忆等。作为 Isabel 的宿主，她还可以人为地为 Chabela 添加更多专业知识和技能，使其更加强大。

让我们把这个区别说得更具体：

Isabel 是一个性格直爽的产品经理，喜欢用短句沟通，讨厌冗长的会议纪要。她的 iFay "Chabela" 写出来的邮件就是 Isabel 的风格——简洁、有力、偶尔带点幽默。当 Chabela 帮 Isabel 拒绝一个不靠谱的需求时，她会用 Isabel 惯用的方式：先肯定对方的出发点，再用数据说明为什么不可行。

而 Milson 是一个温和的工程师，习惯用长段落解释技术细节。如果 Milson 也用同一个 Agent，Agent 不会知道 Milson 喜欢在代码注释里写俳句，不会知道他每次 code review 都会先说"这个思路很有意思"。但 Milson 的 iFay 知道这一切——因为她就是 Milson 的数字化身。

<br>

### Agent vs iFay 对比

| 维度 | Agent | iFay |
|------|-------|------|
| **本质** | 工具——一个功能强大的应用 | 化身——你在数字世界的另一个自己 |
| **人格** | 无个性，所有用户体验相同 | 独特人格，继承宿主的性格、偏好和风格 |
| **记忆** | 会话级记忆，用完即弃 | 终身记忆，与宿主共同成长 |
| **成长** | 版本更新，所有用户同步变化 | 个性化成长，每个 iFay 的成长轨迹独一无二 |
| **归属** | 属于服务提供商 | 属于宿主本人 |
| **宿主过世后** | 账号注销，数据清除 | 人格延续，可由监护人接管或在数字墓园中继续存在 |
| **协作方式** | API 调用 | 心灵感应——语义级直接通信，无信息损失 |
| **与硬件的关系** | 通过 App 间接控制 | 直接接管，硬件是 iFay 的"肢体" |

一句话总结：**Agent 是你雇的员工，iFay 是你自己的分身。**

# 💡 iFay 框架

iFay 是一个可运行的智能代理实例，需要 3 + 1 个核心技术层才能有效运行。我们称之为 CPE + M 框架，自底向上构建：
- _**上下文（Context, C）**_：iFay 感知和行动的外部环境。
- _**协议（Protocol, P）**_：统一的结构化语义定义，使软件开发者、硬件制造商和 Fay 训练师能够无缝协作，无需定制点对点集成。
- _**环境（Environment, E）**_：概念上类似于 Docker。任何 Fay——无论其开发语言——都可以被打包为标准容器，并在 FayGer 运行时（类似 JRE 风格的虚拟环境）中跨平台和跨设备执行。这使得 Fay 可以嵌入任何软件或硬件。

为了激励人类和 Fay 做出有价值的贡献，还有一个贯穿所有三层的第四层：
- _**贡献度量（Merit, M）**_：全球贡献链（Global Merit Chain）追踪、衡量和评估贡献，并以 MeriToken 奖励贡献者。贡献不限于 iFay 和 coFay——还包括提供信息组装服务、API、设备、运行时环境或任何其他被认可的增值输入。

**只需一个声明文件就能组装你的 iFay**：FayManifest 是 iFay 的声明式组装配置，类似 `package.json`。你只需要声明"我需要哪些部件和协议"，FayGer 运行时会自动解析依赖、补充基础设施、组装实例。一个控制无人机的 iFay，Manifest 可能只有 20 行 JSON。

![iFay 框架 Alpha 1](./illustration/ifay-framework-alpha1.png)

iFay 本身由 6 个核心组件组成，分为 4 层（如上图橙色部分所示）：
- 社交层（Social Layer）
- 交互层（Interaction Layer）
- 认知层（Cognition Layer）
- 自我层（Ego Layer）

# 🧭 设计原则

以下五条原则贯穿 iFay 的整个设计和实现，是 iFay 被生态伙伴和用户接受的关键。

### 原则 1：渐进式采纳（Progressive Adoption）
iFay 的生态伙伴不需要实现 100% 符合所有标准的完整 iFay 才能发布产品。一个只用于控制无人机的 iFay，只需要满足其所需的部件子集即可上线。
> 🎯 场景：一家无人机公司想让用户的 iFay 操控他们的产品。他们不需要实现整个 iFay 规范——只需要 CAP 协议 + 设备驱动中枢 + Ego 模型，写一个 FayManifest，周末就能跑起来。

### 原则 2：声明式极简组装（Declarative Minimal Assembly）
iFay 的组装方案必须极致简单——几乎只需要在一个声明文件（FayManifest）中声明所需的部件、协议和配置即可。
> 🎯 场景：开发者打开编辑器，写一个 JSON 文件声明"我需要 CAP 协议 + 传感器 + 飞行控制技能"，运行 `fayger assemble`，一个能飞无人机的 iFay 实例就组装好了。

### 原则 3：灵活的部件组合（Flexible Composition）
部件之间松耦合，可以自由组合。不同厂商的部件可以混搭，只要符合接口契约。
> 🎯 场景：你用 A 厂商的 Ego 模型、B 厂商的语音传感器、C 厂商的设备驱动——它们来自不同公司，但因为都遵循 iFay 接口标准，可以无缝协作。

### 原则 4：人格化而非工具化（Personified, Not Toolified）
iFay 与 Agent 的根本区别：Agent 是工具，iFay 是人格化身。每个 iFay 都有独特的个性、记忆和偏好，是宿主的增强版。
> 🎯 场景：你让 iFay 帮你写一封拒绝邀请的邮件。Agent 会写出一封礼貌但千篇一律的模板；你的 iFay 会用你的口吻写——因为她知道你跟这个人关系不错，所以会加一句"下次一定，这次真走不开"。

### 原则 5：场景驱动的直觉设计（Scenario-Driven Intuition）
产品文档和设计必须让读者能直观想象有了 iFay 后的生活和工作场景，而不是堆砌技术概念。
> 🎯 场景：你读到"第一人称追踪器"这个模块名，可能一头雾水。但如果我说"它就是 iFay 的眼睛——她看到的和你屏幕上看到的一模一样"，你立刻就懂了。

# 🏗️ 四层架构详解

## 🤝 社交层
之所以称为社交层，是因为它管理着 iFay 与人类、设备、资源和资产的关系。

该层定义了 iFay 被授权做什么。
在项目早期阶段（参见[路线图第一阶段](./Roadmap:-5-steps)），其实现仅聚焦于[凭证管理](./Credentials-Management)。

> 🎯 场景：你把淘宝账号委托给 iFay。iFay 拿到的不是你的原始密码，而是一个安全副本凭证。她能用这个副本帮你下单，但如果副本泄露，你可以一键撤销，原始密码不受影响。这就是社交层在做的事——管理信任关系。

<br>

## 🖱 交互层
该层是 iFay 与外部世界的接口。

就像人体一样，它允许 iFay 对环境进行操作和感知。

因此，它由两个主要组件组成：

### 感知（Sense）
你可以把这一层看作 iFay 的感觉系统——它的眼睛、耳朵、触觉和情绪状态。
为此，我们至少需要 3 个核心模块：
- _**[第一人称追踪器（First-person Tracer）](./First-person-Tracer)**_：模拟宿主的第一人称视角——例如，宿主在屏幕或界面上看到的内容。
- _**[传感器（Sensor）](./Sensor)**_：类似于人类神经系统的广义概念，但覆盖范围更广，能够与任何外部传感器集成。
- _**[自我感知（Self-awareness）](./Self-awareness)**_：第一人称追踪器向外看，而这个模块向内看——监测宿主的反应以推断意图，就像一个熟练的助手解读老板的面部表情。

> 🎯 场景：你盯着屏幕上一个复杂的表格皱了皱眉。第一人称追踪器看到了表格内容，自我感知模块捕捉到了你的皱眉——iFay 判断你可能需要帮助理解这些数据，于是主动生成了一份可视化摘要。

### 动作（Action）
你可以把这一层看作 iFay 的运动系统——它的手、脚、嘴等。通过它，iFay 可以控制硬软件。它至少包含三个模块：
- _**[模拟操作（Simulated Operation）](./Simulated-Operation)**_：模拟人类宿主的操作，确保 iFay 在需要时能像人一样操作传统界面。
- _**[技能调用（Invoke Skill）](./Invoke-Skill)**_：直接触发特定技能或执行任务，类似于函数调用或 API 调用。
- _**[自驱行为（Self-driven Behavior）](./Self-driven-Behavior)**_：代表无工具动作，如跑步或俯卧撑——类似于系统设计中的定时任务或时间触发操作。

> 🎯 场景：你让 iFay 帮你在一个老旧的政府网站上填表。这个网站没有 API，没有 iFay 适配——没关系。模拟操作模块让 iFay 像人一样点击、输入、提交，通过第一人称追踪器的视觉反馈自适应地探索界面。不需要脚本，不需要网站改造。

<br>

## 🧠 认知层
该层定义了 iFay 理解什么、记住什么、知道什么以及能做什么。

### 思维（Thought）
该层代表 iFay 的认知能力。它包含宿主用户的数据和 iFay 的个人数据，作为长期持久记忆。其核心模块包括[个人数据堆](./Personal-Data-Heap)、[外部知识](./External-Knowledge)和[对齐意识](./Aligned-Consciousness)。

它还包括外部知识源——可以把这些看作一个人应该知道但已经忘记或从未完全学过的信息。iFay 提供了恢复和整合这些知识的机制。

> 🎯 场景：你三年前去过一次京都，拍了很多照片，写了一些笔记。现在朋友问你推荐京都的餐厅，你已经记不清了。但你的 iFay 记得——她从个人数据堆中找到了你当时的照片、点评和定位，甚至结合你现在的口味偏好变化，给出了更新后的推荐。

### 技能（Skill）
代表能力、专业知识和权限。其核心模块包括[设备驱动中枢](./Device-Driver-Hub)、[注册技能](./Registered-Skill)和[内部技能](./Internal-Skill)。

重要区分：
- 技能（Skill）= iFay 能做什么
- 动作（Action）= iFay 实际在做什么

> 🎯 场景：你的 iFay 注册了六种类型的技能——API、工作流、Bot、Agent、APP、微服务。她知道怎么调用翻译 API、怎么执行报销工作流、怎么操作公司内部的审批 Bot。技能是她的"能力清单"，动作是她此刻正在做的事。

<br>

## 🧬 自我层
该层负责塑造 iFay 的个性。

Ego 模型是一个内嵌的微型模型，独立于外部大型模型运行。它约束 iFay 在价值取向、兴趣偏好、习惯、认知边界、技能边界、权限边界和工作风格等维度与宿主对齐。即使断网，Ego 模型也能在本地运行，确保 iFay 的人格不会因为网络中断而"失忆"。

Ego 是可插拔、可切换的。宿主可能有多面人格——工作中严谨专业，朋友聚会时轻松幽默。iFay 支持多个 Ego 版本，根据场景手动或自动切换，但任一时刻只有一个活跃人格，不会"人格分裂"。

> 🎯 场景：你在公司开会时，iFay 用的是"职场版"Ego——措辞正式、逻辑严密、数据驱动。下班后你跟朋友聊天，iFay 自动切换到"生活版"Ego——语气轻松、偶尔开玩笑、会用表情包。同一个 iFay，不同的面。

详情请参阅 [Ego 模型](https://github.com/ChainModePilot/iFay/wiki#5-ego-model)。
