# 8.1 凭证管理

## 一句话定义

凭证管理是 iFay 的**钥匙串**——它安全地存放着 iFay 代替你使用各种服务所需的"钥匙"。但这些钥匙不是你的原始钥匙，而是安全副本——即使副本丢了，你的原始钥匙依然安全。

## 为什么需要它

想想你的日常生活：你有家门钥匙、车钥匙、办公室门禁卡、银行卡密码、各种 APP 的登录账号……这些都是"凭证"——证明"你有权使用某个东西"的凭据。

现在你想让 iFay 代替你做事——帮你登录电商平台买东西、帮你用公司邮箱发邮件、帮你操控智能家居。iFay 要做这些事，就需要你的"钥匙"。但你会把家门钥匙的原件交给一个助手吗？

更聪明的做法是：配一把**副本钥匙**给助手。副本能开门，但如果副本丢了，你可以立刻作废它，而你的原始钥匙不受影响。这就是凭证管理的核心思路——**副本机制**。

iFay 的凭证管理不只是一个密码管理器。它管理七种不同类型的凭证，为每种凭证创建安全副本，标记每个凭证的所有者（是你的还是 iFay 自己的），记录每次使用的审计日志，并在检测到异常时自动撤销副本。它是 iFay 安全体系的基石。

## 它在架构中的位置

```
iFay 四层架构
├── 社交层          ← 凭证管理在这里
│   ├── FayID              ← 凭证类型之一，也是 iFay 的身份标识
│   └── 👉 凭证管理         ← iFay 的"钥匙串"
├── 交互层
│   └── 动作（Action）
│       ├── 模拟操作        ← 模拟操作时需要凭证登录应用
│       ├── 技能调用        ← 调用技能时需要凭证认证
│       └── ...
├── 认知层
│   └── 技能（Skill）
│       └── 注册技能        ← 注册技能的预授权依赖凭证
└── 自我层
    └── Ego 模型
```

凭证管理位于**社交层**，和 FayID 并列。社交层定义的是 iFay 与外部世界的"关系"——你是谁（FayID）、你有权做什么（凭证）。凭证管理向下为交互层和认知层的所有需要认证的操作提供凭证支持。

你可以把社交层想象成一个人的"社会身份"——身份证（FayID）证明你是谁，各种证件和钥匙（凭证）证明你有权做什么。凭证管理就是帮你管理这些证件和钥匙的安全钥匙串。

## 它是怎么工作的

凭证管理的工作方式可以用四个关键词概括：**分类、复制、标记、守护**。

**1. 分类——七种凭证类型**

iFay 管理的凭证分为七种类型（随着迭代可能会添加更多）：

![凭证管理](./illustration/credential-management.png)

- **身份标识（FayID）**：iFay 的全局唯一身份，就像你的身份证号码
- **账号 / 密码**：最常见的凭证类型，用于登录各种应用和服务
- **证书**：数字证书，用于加密通信和身份验证
- **授权**：第三方授权（如 OAuth），允许 iFay 代表你访问特定服务
- **访问令牌**：临时的访问凭据，通常有有效期
- **智能合约**：区块链上的智能合约凭证，用于去中心化服务
- **数字代币（MeriToken）**：对应社会贡献度的数字代币，用于 iFay 生态中的价值交换

**2. 复制——副本机制**

当你把凭证委托给 iFay 时，凭证管理不会直接使用你的原始凭证。它会将原始凭证**交换为对应的副本**，iFay 使用副本进行登录和认证。

这个设计的好处是：
- 如果副本被泄露，你可以立刻撤销它，而原始凭证不受影响
- 副本可以设置更严格的权限范围（比如只允许读取，不允许修改）
- 副本的使用可以被完整追踪和审计

就像你给保姆配了一把只能开大门的钥匙，而你自己的万能钥匙安全地放在保险箱里。

**3. 标记——谁的钥匙**

每种凭证都会标明其原始所有者是**人类原型**还是 **iFay 本身**。

这很重要，因为 iFay 不只是使用你的凭证——它也可以拥有自己的凭证。比如：
- 你的电商平台账号密码 → 所有者是**人类原型**（你委托给 iFay 使用的）
- iFay 自己注册的某个 API 访问令牌 → 所有者是 **iFay**（iFay 自主获得的）
- 通过你的权限换取的某个服务授权 → 来源是**人类原型权限换取**

人类原型权限和 iFay 权限之间没有包含关系——iFay 可以拥有你没有的权限，你也可以拥有没有委托给 iFay 的权限。

**4. 守护——安全防线**

凭证管理不只是存储和使用凭证，它还是一道安全防线：

- **审计追踪**：每次凭证的使用都会被记录，你可以随时查看 iFay 用你的凭证做了什么
- **异常检测**：如果检测到凭证副本被异常使用（比如从未知地点登录），系统会立刻撤销该副本并通知你
- **隐私保护**：当 iFay 被授权查询隐私数据时，它只会返回**布尔结果**（真或假），而不会暴露实际数据

最后一点特别重要。比如有人声称自己的身份证号是 XXXXXX，你让 iFay 去验证。iFay 查询后只会告诉你"是"或"不是"——它不会把那个人的真实身份证号告诉你。这确保了即使 iFay 有权访问敏感数据，数据本身也不会泄露。

## 与其他模块的关系

| 相关模块 | 关系 | 人体类比 |
|---------|------|---------|
| **FayID** | FayID 是七种凭证类型之一，也是 iFay 的核心身份标识 | 身份证是最基础的证件 |
| **注册技能** | 注册技能的预授权依赖凭证管理提供的认证凭证 | 上岗需要出示证件 |
| **模拟操作** | 模拟操作在登录应用时需要使用凭证 | 开门需要钥匙 |
| **技能调用** | 技能调用在执行任务时需要凭证进行认证 | 办事需要带证件 |
| **iFay Profile** | 凭证信息记录在 Profile 的权限维度中 | 证件列表是身份证的一部分 |

## 场景故事

### 场景一：委托电商平台账号——副本机制的安全保障

你想让 iFay 帮你在电商平台上买日用品。你需要把电商平台账号委托给 iFay。

委托过程中，凭证管理做了以下事情：
1. 接收你的电商平台账号密码（原始凭证）
2. 生成一个对应的**副本凭证**——这个副本可以登录你的电商平台账号，但权限被限制为"仅限购买日用品类目，单笔不超过 200 元"
3. 将副本凭证安全存储，标记所有者为"人类原型"
4. 你的原始密码不会被 iFay 直接持有

一个月后，iFay 帮你买了洗衣液、纸巾、垃圾袋……一切正常。但某天，凭证管理检测到一个异常：有人试图用这个副本凭证购买一台 5000 元的电视——这超出了授权范围。

凭证管理立刻：
- 拒绝了这次购买请求
- 撤销了这个副本凭证
- 向你发送通知："你的电商平台凭证副本检测到异常使用，已自动撤销。你的原始账号密码未受影响，建议检查账号安全。"

你的原始密码始终安全。即使副本被泄露，损失也被控制在最小范围内。

### 场景二：隐私查询——只返回真或假

你是一家公司的 HR，需要核实一位求职者提供的学历信息。你让 iFay 帮你查询。

iFay 被授权访问学历验证数据库。它发起查询："学历证书编号 XXXXX 是否有效？"

数据库返回了完整的学历信息——姓名、学校、专业、毕业时间……但凭证管理的隐私保护机制介入了。iFay 只向你反馈一个结果：**"是，该学历证书有效。"**

你不会看到求职者的其他隐私信息，iFay 也不会存储这些信息。查询的目的是验证真伪，而不是获取数据。凭证管理确保了这条边界不会被突破——无论是有意还是无意。

这个设计背后的理念是：**iFay 有权查询，但无权泄露**。它可以帮你确认一个事实，但不会把别人的隐私数据暴露给你。

## 对开发者

凭证管理属于 **阶段 1（模拟人类操作）** 的核心模块，是 iFay 最早需要实现的模块之一。

- **需求编号**：需求 3（凭证管理）
- **接口规范**：`CredentialManager` 接口，包含 `delegate()`（委托凭证/创建副本）、`authenticate()`（使用凭证认证）、`revoke()`（撤销凭证副本）、`queryPrivacy()`（隐私查询，仅返回布尔值）和 `getAuditLog()`（审计日志）五个核心方法
- **七种凭证类型**：`fay_id`、`account_password`、`certificate`、`authorization`、`access_token`、`smart_contract`、`meri_token`
- **凭证所有者**：`host`（人类原型）或 `ifay`（iFay 本身）
- **副本状态**：`active`（活跃）、`revoked`（已撤销）、`expired`（已过期）
- **关联模块**：注册技能（`RegisteredSkillManager`）的预授权依赖凭证，模拟操作（`SimulatedOperation`）和技能调用（`InvokeSkillService`）在执行时使用凭证认证
- **FayID 独立项目**：FayID 作为凭证类型之一，同时也是[独立开源项目](https://github.com/ChainModePilot/FayID)，详见 [FayID 文档](https://github.com/ChainModePilot/FayID/wiki)
- **设计要点**：副本机制隔离原始凭证风险；隐私查询仅返回布尔值；所有操作记录审计日志；异常检测触发自动撤销；人类原型权限与 iFay 权限无包含关系
- **合规性测试**：iFACTS L1 验证七种凭证类型的管理能力和副本机制，L2 验证与注册技能和技能调用的认证接口，L4 验证隐私查询的布尔返回约束和异常检测行为
