# 12.2 Registered Skill

✅ "Registered Skill" refers to a skill that iFay has already mastered or gained the ability to use.

❌ It does not mean registering or creating an account in any external system.

## One-line Definition

Registered Skill is iFay's **skill certification** — just as humans need a driver's license to drive or a medical license to practice medicine, iFay needs to "register" a skill before it can use it. Registration isn't just record-keeping — it's pre-authorization, ensuring no additional authentication is needed at execution time.

## Why It's Needed

Think about your experience going to the hospital. You wouldn't ask a random passerby to perform surgery on you — you need a doctor with a **professional license**. That license means: this person has undergone professional training, passed exams, and obtained a medical permit. With this license, they can treat you directly without having to re-prove their competence every time.

iFay's Registered Skill works the same way. iFay can do many things — call translation APIs, execute workflows, launch a Bot, use an APP… but before doing anything, that skill must first be "registered." The registration process is like obtaining a certification:

- **Confirm capability**: What type is this skill? What can it do?
- **Complete pre-authorization**: Obtain usage permission in advance, so the actual execution doesn't need to go through the authentication process again
- **Record on file**: iFay's Profile records all registered skills, like listing all certifications on your resume

Why is "pre-authorization" so important? Imagine you're an emergency doctor and a patient arrives — you can't say "hold on, let me go get my license first." Pre-authorization ensures iFay can **act immediately** when it needs to execute a task, without delays from authentication processes.

iFay supports six skill types: **API, Workflow, Bot, Agent, APP, Microservice**. Regardless of type, registration is the prerequisite for use — no registration, no execution rights.

## Its Position in the Architecture

```
iFay Four-Layer Architecture
├── Social Layer
│   └── Credential Management      ← Registered Skill pre-authorization depends on credentials
├── Interaction Layer
│   └── Action
│       ├── Invoke Skill           ← Invocation requires the skill to be registered
│       └── Self-driven Behavior   ← Autonomous behavior also needs to call registered skills
├── Cognitive Layer          ← Registered Skill is here
│   ├── Thought
│   │   └── ...
│   └── Skill
│       ├── Device Driver Hub
│       ├── 👉 Registered Skill    ← iFay's "skill certification library"
│       └── Internal Skill         ← Complementary to Registered Skill: externally learned vs innate

```

Registered Skill sits in the **Cognitive Layer's Skill subsystem**, alongside the Device Driver Hub and Internal Skill. It provides the "callable skill inventory" upward to the Interaction Layer's Invoke Skill and Self-driven Behavior, and depends downward on the Social Layer's Credential Management to complete pre-authorization.

Think of it as a person's "certification cabinet" — storing all the certifications you've earned. When you need to do something, first check if there's a corresponding certification in the cabinet; you can only work with a certification, otherwise you need to go earn one first.

## How It Works

Registered Skill's operation can be summarized in four keywords: **Register, Pre-authorize, Cache, Refresh**.

**1. Register — Skill "onboarding"**

When iFay needs to use a new skill, registration must be completed first. The registration process records key skill information:
- Skill name and type (one of six: API, Workflow, Bot, Agent, APP, Microservice)
- Skill's access endpoint (where to call it)
- Authentication method the skill requires
- Skill's capability description (what it can do)

Once registration is complete, the skill is recorded in iFay's Profile, becoming one of the things iFay "can do."

**2. Pre-authorize — Getting the "pass" in advance**

The core value of registration lies in pre-authorization. During registration, iFay completes all authentication steps in advance — obtaining access tokens, establishing secure connections, verifying permissions. This way, when a task actually needs to be executed, iFay can call it directly with zero delay.

This is like getting your visa in advance — at the airport you board directly, no waiting in line at customs for approval.

**3. Cache — Works offline too**

iFay isn't always online. When iFay is offline (e.g., you're on a plane), the Registered Skill module caches pending actions. Once the network recovers, these cached actions are executed asynchronously.

This is like writing ten emails on the plane — once you land and your phone connects to the network, the emails are sent automatically.

**4. Refresh — Keeping authorization valid**

Pre-authorization isn't permanent. Access tokens expire, permissions may change. The Registered Skill module periodically refreshes authorization status, ensuring each skill's permissions are valid when called. If a skill's authorization expires, the module automatically attempts renewal; if renewal fails, it marks the skill as "pending re-authorization."

This is like your driver's license needing periodic renewal — expired means you can't drive, but once renewed you're back on the road.

## Scenario Stories

### Scenario 1: Registering a translation API — instant response from pre-authorization

You frequently need to communicate with overseas clients, so you have iFay register a professional translation API skill.

During registration, iFay completes the following steps:
- Records the translation API's type (API-type skill), endpoint address, and capability description
- Uses your delegated credentials to complete API authentication and obtain an access token
- Securely stores the token and marks pre-authorization status as "completed"

A week later, you receive an email in Japanese. You tell iFay: "Help me translate this email."

Because the translation skill was already registered and pre-authorized, iFay **immediately** calls the translation API, presenting the translation result to you in two seconds. No popup asking for your password, no spinning wheel waiting for authentication — everything was prepared during registration.

Without pre-authorization? iFay would need to first apply for API access, wait for approval, obtain a token… by the time all that's done, you might have already translated it yourself using translation software. The value of pre-authorization: **front-loading preparation so execution is frictionless**.

### Scenario 2: Offline caching on a plane

You're on a flight to New York, phone in airplane mode. During the flight, you think of several things you need iFay to handle:

- "Send an email to Director Zhang confirming next Wednesday's meeting time"
- "Book the highest-rated Japanese restaurant near my New York hotel"
- "Check tomorrow's weather in New York"

iFay receives these three requests. Sending email requires the email API skill, booking a restaurant requires the dining reservation skill, checking weather requires the weather query skill — all three skills are already registered, but there's no network right now.

The Registered Skill module caches these three pending actions and tells you: "Got it. These three things require internet access to complete. I'll handle them immediately after landing."

The plane lands, you turn on your phone, and the network recovers. The Registered Skill module detects the restored connection and immediately executes the three cached actions asynchronously: email sent, restaurant booked, weather information pushed to you. The entire process doesn't even require you to remind iFay again — it remembers everything you said on the plane.

## For Developers

Registered Skill belongs to **Phase 2 (Direct Client Takeover)** as a core module.

- **Requirement ID**: Requirement 9 (Registered Skill Management)
- **Interface Spec**: `RegisteredSkillManager` interface with five core methods: `register()` (register skill), `query()` (query skill), `refreshAuthorization()` (refresh authorization), `cacheOfflineAction()` (cache offline action), and `flushCachedActions()` (execute cached actions)
- **Six Skill Types**: `api`, `workflow`, `bot`, `agent`, `app`, `microservice`
- **Pre-authorization States**: `pre_authorized`, `pending`, `expired`
- **Associated Modules**: Invoke Skill (`InvokeSkillService`) consumes registered skills, Credential Management (`CredentialManager`) provides authentication support, Internal Skill (`InternalSkill`) provides complementary capabilities
- **Design Notes**: Registration is the prerequisite for any action; pre-authorization is completed during registration ensuring zero-delay execution; offline caching supports asynchronous execution; authorization status is periodically refreshed
- **Conformance Testing**: iFACTS L1 validates registration and pre-authorization for all six skill types, L2 validates interface integration with Invoke Skill and Credential Management, L3 validates the end-to-end offline caching and recovery execution flow
