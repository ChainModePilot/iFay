# 12.2 登録スキル

## 一言定義

登録スキルはiFayの**スキル証明書**——人間が運転免許を取得しないと車を運転できないように、iFayもスキルを「登録」しないと使用できない。登録は単なる記録ではなく、事前認可——実行時に追加の認証が不要となることを保証する。

## なぜ必要か

「事前認可」が重要な理由：救急医のように、患者が運ばれてきた時に「ちょっと待って、先に免許を取りに行く」とは言えない。事前認可はiFayがタスク実行時に**即座に行動**でき、認証フローで遅延しないことを保証する。

iFayは6種類のスキルタイプをサポート：**API、ワークフロー（Workflow）、Bot、Agent、APP、マイクロサービス（Microservice）**。どのタイプでも、登録が使用の前提——登録なしに実行権なし。

## アーキテクチャ内の位置

```
iFay 4層アーキテクチャ
├── ソーシャルレイヤー
│   └── クレデンシャル管理    ← 登録スキルの事前認可はクレデンシャルに依存
├── インタラクションレイヤー
│   └── アクション（Action）
│       ├── スキル呼び出し    ← 呼び出しの前提はスキルが登録済みであること
│       └── 自律行動          ← 自律行動も登録済みスキルを呼び出す
├── コグニションレイヤー    ← 登録スキルはここ
│   └── スキル（Skill）
│       ├── デバイスドライバーハブ
│       ├── 👉 登録スキル      ← iFayの「スキル証明書庫」
│       └── 内部スキル        ← 登録スキルと相互補完
└── エゴレイヤー
```

## 動作の仕組み

**1. 登録——スキルの「入社手続き」**
スキルの名称、タイプ、アクセスエンドポイント、認証方式、能力記述を記録する。

**2. 事前認可——事前に「通行証」を取得**
登録プロセスですべての認証ステップを事前完了——アクセストークン取得、セキュア接続確立、権限検証。

**3. キャッシュ——オフラインでも動作**
iFayがオフライン時、実行待ちのアクションをキャッシュし、ネットワーク回復後に非同期実行する。

**4. リフレッシュ——認可の有効性を維持**
認可状態を定期的にリフレッシュし、各スキルの権限が有効であることを確保する。

## 他モジュールとの関係

| 関連モジュール | 関係 | 人体の類比 |
|---------|------|---------|
| **スキル呼び出し** | スキル呼び出しは登録済みスキルのみ呼び出し可能 | 資格証があって初めて就業可能 |
| **クレデンシャル管理** | 事前認可はクレデンシャル管理の認証クレデンシャルに依存 | 資格証の申請には身分証明が必要 |
| **内部スキル** | 登録スキルは外部から学んだ能力、内部スキルは生来の能力、相互補完 | 取得した証書 vs 生まれ持った直感 |

## シナリオストーリー

### シナリオ1：翻訳API登録——事前認可による即時応答

翻訳APIスキルを登録すると、登録時にAPIキーのバインディングと権限検証が完了する。1週間後に日本語メールを受信した時、iFayは**即座に**翻訳APIを呼び出し、2秒で翻訳結果を提示する。

### シナリオ2：飛行機内のオフラインキャッシュ

飛行中に3つのリクエストを出す。登録スキルモジュールが3つの実行待ちアクションをキャッシュし、着陸後にネットワーク回復を検出して即座に非同期実行する。

## 開発者向け

登録スキルは**フェーズ2（クライアントの直接接管）**のコアモジュール。

- **要件番号**：要件9（登録スキル管理）
- **インターフェース仕様**：`RegisteredSkillManager`インターフェース、`register()`、`query()`、`refreshAuthorization()`、`cacheOfflineAction()`、`flushCachedActions()`の5つのコアメソッド
- **6種類のスキルタイプ**：`api`、`workflow`、`bot`、`agent`、`app`、`microservice`
- **事前認可状態**：`pre_authorized`、`pending`、`expired`
- **コンプライアンステスト**：iFACTS L1で6種類のスキルタイプの登録と事前認可能力を検証、L2でスキル呼び出しとクレデンシャル管理とのインターフェース対接を検証、L3でオフラインキャッシュと復帰実行のエンドツーエンドフローを検証
