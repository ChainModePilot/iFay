# 12.1 デバイスドライバーハブ

## 一言定義

デバイスドライバーハブはiFayの**運動神経**——ハードウェアがiFayの「肢体」なら、デバイスドライバーハブは脳と肢体を接続する運動神経である。iFayが各種ハードウェアデバイスを正しく制御でき、新デバイス接続時に「歩き方を再学習」する必要がないことを保証する。

## なぜ必要か

デバイスドライバーハブは**ハブ層**（Hub）であり、特定のドライバーでもドライバーの単純な集合でもない。標準化インターフェースを提供し、新しいデバイスドライバーが「プラグアンドプレイ」で接続でき、iFayの他のモジュールは変化を全く感知しない。

## アーキテクチャ内の位置

```
iFay 4層アーキテクチャ
├── コグニションレイヤー    ← デバイスドライバーハブはここ
│   └── スキル（Skill）
│       ├── 👉 デバイスドライバーハブ  ← すべてのハードウェアの「運動神経」
│       ├── 登録スキル
│       └── 内部スキル
└── エゴレイヤー
```

CAP（Control Authority Protocol）を通じて端末ハードウェアのドライバー、ローカルインターフェース、コマンドを直接呼び出す。

## 動作の仕組み

**1. 標準化——統一のインターフェース契約**
すべてのデバイスドライバーが同一のインターフェース仕様に従う：ドライバー登録、デバイス呼び出し、状態照会。

**2. 透明化——他モジュールに完全に透明**
新しいデバイスドライバーが追加されても、iFayの他のモジュールは一切修正不要。

**3. グレースフルデグレード——ドライバー失敗でiFayがクラッシュしない**
デバイスドライバーのロード失敗やデバイスオフライン時：状態を報告し、デグレード操作を提案し、他のデバイスの制御は影響を受けない。

## 他モジュールとの関係

| 関連モジュール | 関係 | 人体の類比 |
|---------|------|---------|
| **センサー** | センサーの実際のハードウェアインターフェースを管理 | 神経末端の信号が運動神経を通じて伝導 |
| **スキル呼び出し** | ハードウェア関連タスク実行時にデバイスドライバーハブを通じてデバイスを操作 | 脳が指令 → 運動神経が実行 |
| **パーソナルデータヒープ** | デバイスが生成したデータをデバイスドライバーハブが収集し保存 | 感覚信号 → 記憶保存 |
| **CAPプロトコル** | デバイスドライバーハブがCAPプロトコルでハードウェア制御権限を取得 | 運動神経の「認可チャネル」 |

## シナリオストーリー

### シナリオ1：スマートホーム制御

「映画モードにして」と言うと、iFayがデバイスドライバーハブを通じて同時に複数デバイスを操作：照明を暗く、エアコンを静音モードに、カーテンを閉じ、プロジェクターを起動。4つのデバイスは異なるブランド・異なる通信プロトコルだが、iFayはこれらの差異を知る必要がない。

### シナリオ2：新しいロボット掃除機のプラグアンドプレイ

新しいロボット掃除機をネットワークに接続すると、デバイスドライバーハブが自動的にドライバーをロードし、iFayは即座に制御能力を獲得する。他のモジュールは一切修正不要。

## 開発者向け

デバイスドライバーハブは**フェーズ2（クライアントの直接接管）**のコアモジュール。

- **要件番号**：要件8（デバイスドライバーハブ）
- **インターフェース仕様**：`DeviceDriverHub`インターフェース、`registerDriver()`、`invokeDevice()`、`getDriverStatus()`、`unregisterDriver()`の4つのコアメソッド
- **ドライバー状態**：`loaded`、`active`、`error`、`unavailable`
- **関連プロトコル**：CAP（Control Authority Protocol）
- **設計要点**：ハブ層であり単一ドライバーやドライバー集合ではない；標準化インターフェースで新ドライバー統合が他モジュールに透明；ドライバーロード失敗時はシステムクラッシュではなくデグレード提案を提供
- **コンプライアンステスト**：iFACTS L1でドライバー登録と呼び出し能力を検証、L2でセンサーとパーソナルデータヒープとのインターフェース対接を検証、L4でドライバー失敗時のデグレード行動を検証
