# 8.1 クレデンシャル管理

## 一言定義

クレデンシャル管理はiFayの**キーチェーン**——iFayがあなたの代わりに各種サービスを利用するために必要な「鍵」を安全に保管する。ただしこれらの鍵はあなたの元の鍵ではなく、安全な副本である——副本を紛失しても、あなたの元の鍵は安全なままである。

## なぜ必要か

日常生活を考えてみよう：家の鍵、車の鍵、オフィスの入館カード、銀行カードの暗証番号、各種APPのログインアカウント……これらはすべて「クレデンシャル」——「あなたが何かを使用する権利がある」ことを証明する証拠である。

今、iFayにあなたの代わりに仕事をさせたい——ECプラットフォームにログインして買い物、会社のメールで送信、スマートホームの操作。iFayがこれらを行うには、あなたの「鍵」が必要だ。しかし家の鍵の原本を助手に渡すだろうか？

より賢い方法は：助手に**副本の鍵**を作ること。副本でドアは開けられるが、副本を紛失しても即座に無効化でき、元の鍵は影響を受けない。これがクレデンシャル管理の核心的な考え方——**副本メカニズム**である。

iFayのクレデンシャル管理は単なるパスワードマネージャーではない。7種類の異なるクレデンシャルを管理し、各クレデンシャルの安全な副本を作成し、各クレデンシャルの所有者（あなたのものかiFay自身のものか）を標記し、使用のたびに監査ログを記録し、異常検出時に自動的に副本を取り消す。iFayのセキュリティ体系の礎石である。

## アーキテクチャ内の位置

```
iFay 4層アーキテクチャ
├── ソーシャルレイヤー    ← クレデンシャル管理はここ
│   ├── FayID              ← クレデンシャルタイプの一つ、iFayの身分識別でもある
│   └── 👉 クレデンシャル管理  ← iFayの「キーチェーン」
├── インタラクションレイヤー
│   └── アクション（Action）
│       ├── シミュレーテッドオペレーション  ← シミュレーション操作時にクレデンシャルでログイン
│       ├── スキル呼び出し                ← スキル呼び出し時にクレデンシャルで認証
│       └── ...
├── コグニションレイヤー
│   └── スキル（Skill）
│       └── 登録スキル        ← 登録スキルの事前認可はクレデンシャルに依存
└── エゴレイヤー
    └── Egoモデル
```

クレデンシャル管理は**ソーシャルレイヤー**に位置し、FayIDと並列する。ソーシャルレイヤーが定義するのはiFayと外部世界の「関係」——あなたは誰か（FayID）、あなたは何をする権利があるか（クレデンシャル）。クレデンシャル管理はインタラクションレイヤーとコグニションレイヤーの認証が必要なすべての操作にクレデンシャルサポートを提供する。

## 動作の仕組み

クレデンシャル管理の動作方式は4つのキーワードで要約できる：**分類、複製、標記、守護**。

**1. 分類——7種類のクレデンシャルタイプ**

![クレデンシャル管理](./illustration/credential-management.png)

- **身分識別（FayID）**：iFayのグローバル一意身分
- **アカウント / パスワード**：最も一般的なクレデンシャルタイプ
- **証明書**：デジタル証明書、暗号化通信と身分検証用
- **認可**：サードパーティ認可（OAuthなど）
- **アクセストークン**：一時的なアクセス証拠、通常有効期限あり
- **スマートコントラクト**：ブロックチェーン上のスマートコントラクトクレデンシャル
- **デジタルトークン（MeriToken）**：社会貢献度に対応するデジタルトークン

**2. 複製——副本メカニズム**

クレデンシャルをiFayに委託する際、クレデンシャル管理は元のクレデンシャルを直接使用しない。元のクレデンシャルを**対応する副本に交換**し、iFayは副本を使用してログインと認証を行う。

**3. 標記——誰の鍵か**

各クレデンシャルには元の所有者が**ヒューマンプライム（Human Prime）**か**iFay自身**かが明記される。ヒューマンプライム（Human Prime）の権限とiFayの権限の間に包含関係はない。

**4. 守護——セキュリティ防衛線**

- **監査追跡**：クレデンシャル使用のたびに記録される
- **異常検出**：副本クレデンシャルの異常使用を検出すると即座に取り消し通知
- **プライバシー保護**：プライバシーデータ照会時は**ブール結果**（真または偽）のみを返す

## 他モジュールとの関係

| 関連モジュール | 関係 | 人体の類比 |
|---------|------|---------|
| **FayID** | 7種類のクレデンシャルタイプの一つであり、iFayのコアアイデンティティ | 身分証は最も基本的な証件 |
| **登録スキル** | 登録スキルの事前認可はクレデンシャル管理の認証クレデンシャルに依存 | 就業には証件の提示が必要 |

| **iFay Profile** | クレデンシャル情報はProfileの権限次元に記録 | 証件リストは身分証の一部 |

## シナリオストーリー

### シナリオ1：ECプラットフォームアカウントの委託

iFayにECプラットフォームで日用品を購入させたい。委託プロセスでクレデンシャル管理は以下を行う：
1. ECプラットフォームのアカウント・パスワード（元のクレデンシャル）を受領
2. 対応する**副本クレデンシャル**を生成——権限は「日用品カテゴリのみ、1回200元以下」に制限
3. 副本クレデンシャルを安全に保存、所有者を「ヒューマンプライム（Human Prime）」と標記

ある日、クレデンシャル管理が異常を検出：誰かがこの副本クレデンシャルで5000元のテレビを購入しようとした——認可範囲を超えている。クレデンシャル管理は即座に購入リクエストを拒否し、副本を取り消し、通知を送信する。元のパスワードは常に安全である。

### シナリオ2：プライバシー照会——真か偽のみを返す

あなたはHRで、求職者の学歴情報を確認する必要がある。iFayが学歴検証データベースに照会する。データベースは完全な学歴情報を返すが、プライバシー保護メカニズムが介入する。iFayはあなたに結果のみをフィードバックする：**「はい、その学歴証書は有効です。」**

設計理念：**iFayは照会する権利があるが、漏洩する権利はない**。

## 開発者向け

クレデンシャル管理は**フェーズ1（人間操作のシミュレーション）**のコアモジュール。

- **要件番号**：要件3（クレデンシャル管理）
- **インターフェース仕様**：`CredentialManager`インターフェース、`delegate()`、`authenticate()`、`revoke()`、`queryPrivacy()`、`getAuditLog()`の5つのコアメソッド
- **7種類のクレデンシャルタイプ**：`fay_id`、`account_password`、`certificate`、`authorization`、`access_token`、`smart_contract`、`meri_token`
- **クレデンシャル所有者**：`host`（ヒューマンプライム（Human Prime））または`ifay`（iFay自身）
- **副本状態**：`active`（アクティブ）、`revoked`（取り消し済み）、`expired`（期限切れ）
- **FayID独立プロジェクト**：FayIDはクレデンシャルタイプの一つであると同時に[独立オープンソースプロジェクト](https://github.com/ChainModePilot/FayID)でもある
- **コンプライアンステスト**：iFACTS L1で7種類のクレデンシャルタイプの管理能力と副本メカニズムを検証、L2で登録スキルとスキル呼び出しの認証インターフェースを検証、L4でプライバシー照会のブール返却制約と異常検出行動を検証
