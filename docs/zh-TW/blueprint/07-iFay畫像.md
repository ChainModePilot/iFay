# 7. iFay 畫像

iFay Profile 是 iFay 的完整「身份證」——一個語義可解釋的六維屬性表。它不僅讓人類能夠識別一個 iFay，也讓系統能夠解析它，更讓 Fay 之間能夠相互認識。

人類原型畫像（對齊意識 / Aligned Consciousness）是 iFay Profile 的子集和輸入源。人類原型畫像描述的是「人類原型是什麼樣的人」，而 iFay Profile 描述的是「這個 iFay 是什麼樣的存在」——它包含了人類原型的人格特徵，但也包含了 iFay 自身的技能、硬體、權限等獨立屬性。

人類原型畫像由[對齊意識](./11.3-對齊意識)模組維護。技能維度由[註冊技能](./12.2-註冊技能)管理，硬體維度由[設備驅動中樞](./12.1-設備驅動中樞)管理，權限維度由[憑證管理](./8.1-憑證管理)管理。

---

## iFay 身份

iFay 身份由 FayID 和元資訊組成。FayID 是全域唯一、人類可讀、機器可讀的持久識別碼，是 iFay 在所有互動場景中被識別和追蹤的基礎。

iFay 身份支持平滑的身份遷移——當 iFay 從個人助手演變為自主社會角色時，FayID 不需要更換，身份自然延續。

---

## Ego 模型

Profile 中的 Ego 維度記錄當前活躍的 Ego 版本引用。

Ego 模型是可插拔、可切換的邊緣小模型，載入到任何終端裝置上即可讓該裝置擁有人類原型的個性。iFay 支持擁有多個 Ego 版本，以適應人類原型的多面人格——但任一時刻只有一個版本處於活躍狀態，防止「人格分裂」。

切換方式有兩種：人類原型手動觸發，或 iFay 基於上下文自動判斷。切換完成後，Profile 中的活躍 Ego 版本引用會同步更新。

### Ego 倫理約束

Ego 切換機制受到以下倫理約束：

- 所有 Ego 版本必須共享同一套核心價值觀——切換的是「風格」，而非「道德底線」
- Ego 切換時必須在互動元資料中正確標註當前活躍的 Ego 版本標識，確保可追溯性
- 禁止建立違反人類原型核心價值觀的 Ego 版本
- 自動切換必須基於可解釋的上下文判斷，不得在人類原型無感知的情況下改變行為風格

### 場景

> 張磊是一名產品經理，他為自己的 iFay 配置了兩個 Ego 版本：「專業模式」和「休閒模式」。
>
> 工作日上午，iFay 自動切換到「專業模式」——回覆郵件措辭嚴謹，安排會議時優先考慮效率，處理需求文件時注重邏輯完整性。
>
> 週末晚上，iFay 偵測到張磊在瀏覽美食推薦，自動切換到「休閒模式」——推薦餐廳時更注重氛圍和口味偏好，語氣也變得輕鬆隨意。
>
> 兩個版本，同一個張磊。

## Faying 思維

Faying 思維是 iFay 的認知資產，包含四種子類型：

| 子類型 | 說明 | 示例 |
|--------|------|------|
| **內容（Content）** | 結構化或非結構化的內容資產 | 人類原型的照片集、撰寫的文章、創作的音樂、錄製的影片 |
| **資料（Data）** | 個人資料 | 健康監測資料、消費記錄、應用使用習慣、位置歷史 |
| **知識庫（Knowledge Base）** | 組織化的知識體系 | 專業領域筆記、學習資料庫、工作經驗總結、行業研究報告 |
| **資訊流（Info Feed）** | 即時資訊訂閱 | 關注的新聞源、行業動態推送、社群媒體更新、股票行情 |

思維維度的資料來源於個人資料堆的初始化和持續積累，是 iFay 理解世界和做出判斷的基礎。

## Faying 技能

Faying 技能是 iFay 的行動能力，包含六種技能類型：

| 類型 | 說明 |
|------|------|
| **API** | 可直接呼叫的程式介面，如天氣查詢 API、翻譯 API |
| **工作流（Workflow）** | 多步驟的自動化流程，如「每週一生成週報並發送給團隊」 |
| **Bot** | 特定場景的對話機器人，如客服 Bot、預約 Bot |
| **Agent** | 具備自主決策能力的智能代理，如投資分析 Agent |
| **APP** | 完整的應用程式，如日曆管理 APP、健康追蹤 APP |
| **微服務（Microservice）** | 獨立部署的服務單元，如圖像識別服務、語音轉文字服務 |

技能註冊是 iFay 執行任何動作的前提條件。已註冊的技能會完成預授權，確保執行時無需額外認證。

## Faying 硬體

硬體是 iFay 的「肢體」。iFay 是人格和認知實體，硬體是它與物理世界互動的載體。

### 三種硬體子類型

| 子類型 | 說明 | 示例 |
|--------|------|------|
| **設備（Device）** | 終端硬體裝置 | 手機、電腦、智慧手錶、無人機、智慧家居裝置 |
| **儲存（Storage）** | 資料儲存資源 | 雲端儲存、本地硬碟、NAS、向量資料庫 |
| **算力（Computing）** | 計算資源 | GPU、雲端計算節點、邊緣計算裝置 |

### 硬體發現機制

當 iFay 遷移到新環境時，終端裝置負責掃描當前環境中可 Faying 的硬體——類似藍牙或 WiFi 掃描周圍裝置的方式。iFay 本身不直接執行硬體掃描，而是根據掃描結果進行能力匹配。

這裡有一個關鍵區分：

- **iFay 能操控哪些硬體**，取決於它的認知能力（註冊技能、內部技能）
- **iFay 能連接到哪些硬體**，取決於當前環境中實際可用的裝置

### 場景

> 想像一個會打羽毛球的人走進一間體育館。
>
> 「會打羽毛球」是他的**技能**——這是他自身的能力，無論走到哪裡都帶著。但他能不能打球，還取決於體育館裡有沒有**球拍**——這是環境提供的硬體。
>
> iFay 也是如此。它可能擁有控制無人機的技能（註冊了飛行控制 API），但如果當前環境中沒有可連接的無人機，這個技能就無法施展。反過來，即使環境中有一台 3D 列印機，如果 iFay 沒有註冊相關的操控技能，它也只能「看到」這台裝置卻無法使用。

## Faying 權限

權限是 iFay 的「關係」——它定義了 iFay 與外部世界的信任邊界。

### 三種權限生命週期

| 生命週期 | 說明 | 示例 |
|----------|------|------|
| **與生俱來（Inherent）** | 建立時即擁有的權限 | 存取人類原型日曆、讀取人類原型偏好設定 |
| **持久（Persistent）** | 長期有效的權限 | 人類原型信箱的讀寫權限、企業內網存取權 |
| **臨時（Ephemeral）** | 需要時申請，用後釋放 | 一次性存取醫療記錄、臨時取得某個 API 的呼叫權 |

### 可擴展的鑑權方式

iFay 的鑑權方式是可擴展的，支持 SSO、OAuth、Fingerprint 等方法。這裡的 Fingerprint 是一個統稱，涵蓋一切可識別身份的標識——面部識別、聲紋、裝置指紋等。不同的指紋類型具有不同的確信力度：面部識別的確信力度可能高於裝置指紋，而多因素組合的確信力度又高於單一因素。

### 場景

> 王芳的 iFay 擁有三種不同生命週期的權限：
>
> **與生俱來**：iFay 建立時就自動獲得了存取王芳日曆的權限——這是每個 iFay 的基本能力，不需要額外申請。
>
> **持久權限**：王芳授權 iFay 長期管理她的工作信箱。iFay 可以閱讀郵件、起草回覆、管理標籤，這個權限一直有效，直到王芳主動撤銷。
>
> **臨時權限**：王芳需要查看去年的體檢報告。她讓 iFay 去醫院系統調取記錄。iFay 臨時申請了一次性的醫療資料存取權限，取得報告後權限自動釋放——下次再需要時，需要重新申請。

## Profile 可見性

Profile 的可見性是一個應用層關注點，由具體的應用場景和互動方式決定。

同一個 iFay Profile，在不同場景下展示不同的維度：在 Fay 間協作時，對方可能只需要看到技能和權限維度；在人類原型管理介面中，所有六個維度都完整可見；在公開社交場景中，可能只展示身份和部分技能資訊。

Profile 本身不定義「誰能看到什麼」——這個決策交給使用 Profile 的應用來做。
