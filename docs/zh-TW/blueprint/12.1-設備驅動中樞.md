# 12.1 設備驅動中樞

## 一句話定義

設備驅動中樞是 iFay 的**運動神經**——如果硬體是 iFay 的「肢體」，設備驅動中樞就是連接大腦和肢體的運動神經。它確保 iFay 能正確控制各種硬體裝置，而且新裝置接入時不需要「重新學習走路」。

## 為什麼需要它

想像一下你的身體：你的大腦想拿起桌上的杯子，它不需要知道手臂肌肉的每一根纖維怎麼收縮——運動神經會把「拿杯子」這個意圖翻譯成精確的肌肉指令。更妙的是，當你第一次學會騎腳踏車時，你的大腦不需要重新學習怎麼走路——新的運動技能是在已有的神經通路上「疊加」的，不會破壞原有的能力。

iFay 面對的硬體世界也是如此。智慧燈泡、空調、窗簾、掃地機器人、無人機……每種裝置都有自己的「語言」和控制方式。如果 iFay 每接入一個新裝置就要改一次內部架構，那就像你每學一個新動作就要重新學走路一樣荒謬。

設備驅動中樞解決的就是這個問題。它是一個**中樞層**（Hub），不是某個具體的驅動程式，也不是驅動程式的簡單集合。它提供一套標準化介面，讓任何新裝置驅動都能「即插即用」地接入，而 iFay 的其他模組完全感知不到變化。就像你的運動神經系統——無論你學會了多少新技能，你的大腦結構不需要改變。

## 它在架構中的位置

```
iFay 四層架構
├── 社交層
├── 互動層
│   ├── 感知（Sense）
│   │   ├── 感測器           ← 感知環境資料，硬體介面由驅動中樞管理
│   │   └── ...
│   └── 動作（Action）
│       ├── 技能調用         ← 呼叫技能時可能需要操控硬體
│       └── ...
├── 認知層          ← 設備驅動中樞在這裡
│   ├── 思維（Thought）
│   │   └── ...
│   └── 技能（Skill）
│       ├── 👉 設備驅動中樞   ← 所有硬體的「運動神經」
│       ├── 註冊技能
│       └── 內部技能
└── 自我層
    └── Ego 模型
```

設備驅動中樞位於**認知層的技能子系統**中。它向上為互動層的感測器和動作模組提供硬體控制能力，向下透過 **CAP（控制權限協議）** 直接呼叫終端硬體的驅動程式、本地介面和命令。

你可以把它想像成脊髓——它不做決策（那是大腦/認知層的事），但所有從大腦到肢體的運動指令都必須經過它。它的存在讓大腦可以專注於「想做什麼」，而不用操心「怎麼讓肌肉動起來」。

## 它是怎麼工作的

設備驅動中樞的工作方式可以用三個關鍵詞概括：**標準化、透明化、優雅降級**。

**1. 標準化——統一的介面契約**

無論是智慧燈泡還是工業機械臂，每個裝置驅動在接入時都必須遵循同一套介面規範。驅動中樞定義了標準的註冊、呼叫和狀態查詢介面：

- **註冊驅動**：新裝置驅動宣告自己的類型、能力和支持的命令
- **呼叫裝置**：透過統一的命令格式控制任何已註冊的裝置
- **查詢狀態**：隨時了解某個驅動是否正常工作

這就像你身體的運動神經系統有統一的訊號傳導方式——無論是控制手指還是控制腿部，神經訊號的「協議」是一樣的。

**2. 透明化——對其他模組完全透明**

當一個新的裝置驅動被添加到 iFay 時，iFay 的其他模組（感測器、技能調用、自驅行為等）完全不需要做任何修改。驅動中樞把所有硬體差異都「吸收」了，對外只暴露統一的介面。

這就像你學會了騎腳踏車——你的視覺系統、平衡系統、決策系統都不需要「升級」，只是運動神經系統多了一條新的通路。

**3. 優雅降級——驅動失敗不會讓 iFay 崩潰**

如果某個裝置驅動載入失敗或者裝置離線了怎麼辦？驅動中樞不會讓整個 iFay 因此癱瘓。它會：
- 向 iFay 報告該驅動不可用的狀態
- 提供降級操作建議（比如「空調驅動離線，建議透過模擬操作使用空調遙控器 APP」）
- 其他裝置的控制完全不受影響

這就像你的一隻手受傷了——你的另一隻手、你的腿、你的嘴巴都還能正常工作。身體不會因為一個部位的問題而全面癱瘓。

## 與其他模組的關係

| 相關模組 | 關係 | 人體類比 |
|---------|------|---------|
| **感測器** | 感測器的實際硬體介面由設備驅動中樞管理 | 神經末梢的訊號透過運動神經傳導 |
| **技能調用** | 技能調用在執行涉及硬體的任務時，透過驅動中樞操控裝置 | 大腦下達指令 → 運動神經執行 |
| **個人資料堆** | 裝置產生的資料透過驅動中樞採集，最終儲存在個人資料堆中 | 感覺訊號 → 記憶儲存 |
| **CAP 協議** | 驅動中樞透過 CAP 協議取得硬體控制權限 | 運動神經的「授權通道」 |
| **註冊技能** | 某些註冊技能需要呼叫硬體能力，透過驅動中樞間接實現 | 學會的技能需要透過運動神經來施展 |

## 場景故事

### 場景一：iFay 控制智慧家居

晚上 8 點，你對 iFay 說：「幫我把家裡調成電影模式。」

iFay 理解了你的意圖，然後透過設備驅動中樞同時操控多個裝置：

- **智慧燈泡驅動**：把客廳燈光調暗到 20%，色溫切換為暖黃色
- **空調驅動**：溫度設定為 24°C，風速調到最低（安靜模式）
- **智慧窗簾驅動**：關閉窗簾，遮擋外部光線
- **投影機驅動**：開啟投影機，切換到 HDMI 輸入源

這四個裝置來自不同品牌，使用不同的通訊協議（WiFi、藍牙、Zigbee），但 iFay 完全不需要知道這些差異。設備驅動中樞把所有複雜性都遮蔽了——對 iFay 來說，它只是發出了四條標準化的命令，就像你的大腦只需要想「拿起杯子」，不需要計算每根手指該彎曲多少度。

### 場景二：新掃地機器人即插即用

你買了一台新的掃地機器人回家。你把它連上家庭網路後，iFay 透過終端裝置發現了這個新硬體。

接下來發生的事情對你來說幾乎是無感的：

1. 終端裝置掃描到新的掃地機器人，將其資訊報告給 iFay
2. 設備驅動中樞自動載入掃地機器人的驅動程式（透過標準化介面註冊）
3. 驅動註冊完成後，iFay 立刻獲得了控制掃地機器人的能力

你不需要在 iFay 裡做任何設定。第二天早上你出門上班時，iFay 的自驅行為模組判斷「人類原型出門了，適合打掃」，透過設備驅動中樞啟動掃地機器人，開始清掃。

整個過程中，iFay 的其他模組——感測器、自我感知、Ego 模型——沒有做任何修改。新裝置的接入就像你的身體長出了一條新的「手臂」，運動神經自動連接好了，大腦直接就能控制它。

## 對開發者

設備驅動中樞屬於 **階段 2（直接接管用戶端）** 的核心模組，依賴 CAP 協議。

- **需求編號**：需求 8（設備驅動中樞）
- **介面規範**：`DeviceDriverHub` 介面，包含 `registerDriver()`（註冊驅動）、`invokeDevice()`（呼叫裝置）、`getDriverStatus()`（查詢狀態）和 `unregisterDriver()`（卸載驅動）四個核心方法
- **驅動狀態**：`loaded`（已載入）、`active`（活躍）、`error`（錯誤）、`unavailable`（不可用）
- **關聯協議**：CAP（控制權限協議）用於直接呼叫終端硬體的驅動程式、本地介面和命令
- **關聯模組**：感測器（`SensorModule`）的硬體介面由驅動中樞管理，個人資料堆（`PersonalDataHeap`）儲存裝置資料
- **設計要點**：驅動中樞是中樞層（Hub），不是單個驅動或驅動集合；標準化介面確保新驅動整合對其他模組透明；驅動載入失敗時提供降級建議而非系統崩潰
- **合規性測試**：iFACTS L1 驗證驅動註冊和呼叫能力，L2 驗證與感測器和個人資料堆的介面對接，L4 驗證驅動失敗時的降級行為
