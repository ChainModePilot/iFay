# 12.2 註冊技能

## 一句話定義

註冊技能是 iFay 的**技能證書**——就像人類需要考取駕照才能開車、需要獲得醫師資格才能行醫，iFay 需要「註冊」一個技能才能使用它。註冊不只是記錄，更是預授權——確保執行時不需要額外認證。

## 為什麼需要它

想想你去醫院看病的經歷。你不會隨便找一個路人幫你做手術——你需要一個有**執業資格證**的醫生。這個資格證意味著：這個人經過了專業培訓、通過了考試、獲得了行醫許可。有了這個證，他可以直接給你看病，不需要每次都重新證明自己的能力。

iFay 的註冊技能就是這個道理。iFay 能做的事情很多——呼叫翻譯 API、執行工作流、啟動一個 Bot、使用一個 APP……但在做任何事之前，這個技能必須先被「註冊」。註冊的過程就像考取資格證：

- **確認能力**：這個技能是什麼類型？能做什麼？
- **完成預授權**：提前獲得使用許可，這樣真正執行時就不需要再走一遍認證流程
- **記錄在案**：iFay 的 Profile 中會記錄所有已註冊的技能，就像你的履歷上列出了所有資格證書

為什麼「預授權」這麼重要？想像你是一個急診醫生，病人送來了，你不可能說「等一下，我先去考個執照」。預授權確保了 iFay 在需要執行任務時能**立刻行動**，不會因為認證流程而延誤。

iFay 支持六種技能類型：**API、工作流（Workflow）、Bot、Agent、APP、微服務（Microservice）**。無論哪種類型，註冊都是使用的前提——沒有註冊，就沒有執行權。

## 它在架構中的位置

```
iFay 四層架構
├── 社交層
│   └── 憑證管理           ← 註冊技能的預授權依賴憑證
├── 互動層
│   └── 動作（Action）
│       ├── 技能調用        ← 呼叫的前提是技能已註冊
│       └── 自驅行為        ← 自主行為也需要呼叫已註冊技能
├── 認知層          ← 註冊技能在這裡
│   ├── 思維（Thought）
│   │   └── ...
│   └── 技能（Skill）
│       ├── 設備驅動中樞
│       ├── 👉 註冊技能      ← iFay 的「技能證書庫」
│       └── 內部技能        ← 與註冊技能互補：外部學來的 vs 與生俱來的
└── 自我層
    └── Ego 模型
```

註冊技能位於**認知層的技能子系統**中，和設備驅動中樞、內部技能並列。它向上為互動層的技能調用和自驅行為提供「可呼叫的技能清單」，向下依賴社交層的憑證管理來完成預授權。

你可以把它想像成一個人的「資格證書櫃」——裡面存放著你考取的所有證書。當你需要做某件事時，先看看櫃子裡有沒有對應的證書；有證書才能上崗，沒有就得先去考一個。

## 它是怎麼工作的

註冊技能的工作方式可以用四個關鍵詞概括：**註冊、預授權、快取、重新整理**。

**1. 註冊——技能的「入職登記」**

當 iFay 需要使用一個新技能時，首先要完成註冊。註冊過程會記錄技能的關鍵資訊：
- 技能名稱和類型（六種之一：API、Workflow、Bot、Agent、APP、Microservice）
- 技能的存取端點（在哪裡呼叫它）
- 技能需要的認證方式
- 技能的能力描述（它能做什麼）

註冊完成後，這個技能就被記錄在 iFay 的 Profile 中，成為 iFay「會做的事」之一。

**2. 預授權——提前拿到「通行證」**

註冊的核心價值在於預授權。在註冊過程中，iFay 會提前完成所有認證步驟——取得存取令牌、建立安全連線、驗證權限。這樣，當真正需要執行任務時，iFay 可以直接呼叫，零延遲。

這就像你提前辦好了簽證——到了機場直接登機，不需要在海關排隊等審批。

**3. 快取——離線也能工作**

iFay 不總是在線的。當 iFay 處於離線狀態時（比如你在飛機上），註冊技能模組會把待執行的動作快取起來。等網路恢復後，這些快取的動作會被非同步執行。

這就像你在飛機上寫好了十封郵件，落地後手機一連上網，郵件就自動發出去了。

**4. 重新整理——保持授權有效**

預授權不是永久的。存取令牌會過期，權限可能變更。註冊技能模組會定期重新整理授權狀態，確保每個技能在被呼叫時權限是有效的。如果某個技能的授權過期了，模組會自動嘗試續期；如果續期失敗，會標記該技能為「待重新授權」。

這就像你的駕照需要定期年審——過期了就不能開車，但年審通過後又可以繼續上路。

## 與其他模組的關係

| 相關模組 | 關係 | 人體類比 |
|---------|------|---------|
| **技能調用** | 技能調用只能呼叫已註冊的技能，註冊是執行的前提 | 有證才能上崗 |
| **憑證管理** | 註冊技能的預授權依賴憑證管理提供的認證憑證 | 資格證需要身份證明來申請 |
| **內部技能** | 註冊技能是外部學來的能力，內部技能是與生俱來的能力，二者互補 | 考取的證書 vs 天生的直覺 |
| **自驅行為** | 自驅行為在自主執行任務時，呼叫的也是已註冊技能 | 自發行動也需要有對應的能力 |
| **iFay Profile** | 所有已註冊技能記錄在 Profile 的技能維度中 | 履歷上的資格證書列表 |

## 場景故事

### 場景一：註冊翻譯 API——預授權帶來的即時回應

你經常需要和海外客戶溝通，於是讓 iFay 註冊一個專業翻譯 API 技能。

註冊過程中，iFay 完成了以下步驟：
- 記錄翻譯 API 的類型（API 類技能）、端點位址和能力描述
- 使用你委託的憑證完成 API 的認證，取得存取令牌
- 將令牌安全儲存，標記預授權狀態為「已完成」

一週後，你收到一封日語郵件。你對 iFay 說：「幫我翻譯一下這封郵件。」

因為翻譯技能已經註冊並預授權過了，iFay **立刻**呼叫翻譯 API，兩秒鐘後就把翻譯結果呈現給你。沒有彈窗要求你輸入密碼，沒有等待認證的轉圈圈——一切都在註冊時就準備好了。

如果沒有預授權呢？iFay 就得先去申請 API 存取權限、等待審批、取得令牌……等這些都完成了，你可能已經自己用翻譯軟體翻完了。預授權的價值就在於：**把準備工作前置，讓執行時刻零摩擦**。

### 場景二：飛機上的離線快取

你在飛往紐約的航班上，手機處於飛航模式。途中你想起幾件事需要 iFay 幫忙處理：

- 「幫我給張總發一封郵件，確認下週三的會議時間」
- 「幫我預訂紐約飯店附近評分最高的日料餐廳」
- 「幫我查一下明天紐約的天氣」

iFay 接收了這三個請求。發郵件需要呼叫郵件 API 技能，訂餐廳需要呼叫餐飲預訂技能，查天氣需要呼叫天氣查詢技能——這三個技能都已經註冊過了，但現在沒有網路。

註冊技能模組把這三個待執行的動作快取起來，並告訴你：「好的，我記下了。這三件事需要連網才能完成，落地後我會立刻處理。」

飛機落地，你打開手機，網路恢復。註冊技能模組偵測到連線恢復，立刻非同步執行快取的三個動作：郵件發出了、餐廳預訂好了、天氣資訊也推送給你了。整個過程你甚至不需要再提醒 iFay——它記得你在飛機上說過的每一件事。

## 對開發者

註冊技能屬於 **階段 2（直接接管用戶端）** 的核心模組。

- **需求編號**：需求 9（註冊技能管理）
- **介面規範**：`RegisteredSkillManager` 介面，包含 `register()`（註冊技能）、`query()`（查詢技能）、`refreshAuthorization()`（重新整理授權）、`cacheOfflineAction()`（快取離線動作）和 `flushCachedActions()`（執行快取動作）五個核心方法
- **六種技能類型**：`api`、`workflow`、`bot`、`agent`、`app`、`microservice`
- **預授權狀態**：`pre_authorized`（已預授權）、`pending`（待授權）、`expired`（已過期）
- **關聯模組**：技能調用（`InvokeSkillService`）消費已註冊技能，憑證管理（`CredentialManager`）提供認證支持，內部技能（`InternalSkill`）提供互補能力
- **設計要點**：註冊是任何動作的前提條件；預授權在註冊時完成，確保執行時零延遲；離線快取支持非同步執行；授權狀態定期重新整理
- **合規性測試**：iFACTS L1 驗證六種技能類型的註冊和預授權能力，L2 驗證與技能調用和憑證管理的介面對接，L3 驗證離線快取和恢復執行的端到端流程
