# 10.1 模擬操作

## 一句話定義

模擬操作是 iFay 的**雙手**——它能像人一樣點擊、拖曳、捲動、打字，操作任何現有的軟體介面，不需要軟體專門為 AI 做任何改造。

## 為什麼需要它

想像你請了一個新助手，你希望他幫你處理各種電腦和手機上的事務。但問題是：你用的那些軟體——銀行網站、保險系統、政府辦事平台——它們都是給人類設計的，沒有一個提供「AI 專用入口」。

怎麼辦？最直接的方式就是：**讓助手像你一樣操作這些軟體**。看到按鈕就點擊，看到表單就填寫，看到頁面就捲動。

這就是模擬操作的意義。它讓 iFay 不需要等待世界上所有的軟體都為 AI 重新設計，**現在就能幫你幹活**。

但模擬操作和傳統的「自動化腳本」（比如 RPA）有本質區別。RPA 就像一個只會背台詞的演員——它按照預先寫好的腳本一步步執行，介面稍有變化就會卡住。而 iFay 的模擬操作是**感知驅動**的：它透過第一人稱追蹤器「看」螢幕，理解當前介面的狀態，然後決定下一步該怎麼做。就像一個真正的人類使用者——即使網站改版了，你也能憑直覺找到新的按鈕位置。

## 它在架構中的位置

```
iFay 四層架構
├── 社交層
├── 互動層          ← 模擬操作在這裡
│   ├── 感知（Sense）
│   │   ├── 第一人稱追蹤器   ← 眼睛：看螢幕上有什麼
│   │   ├── 感測器           ← 神經系統：感知環境
│   │   └── 自我感知         ← 內心：讀懂你的意圖
│   └── 動作（Action）
│       ├── 👉 模擬操作      ← 雙手：點擊、拖曳、打字
│       ├── 技能調用         ← 嘴巴：直接「打電話」給服務
│       └── 自驅行為         ← 自主意志：決定何時行動
├── 認知層
└── 自我層
```

模擬操作位於**互動層的動作子系統**中，是 iFay 與外部世界互動的三種方式之一。它和第一人稱追蹤器的關係最為緊密——追蹤器是「眼睛」，模擬操作是「手」，兩者配合實現**手眼協調**。

這就像你用手操作滑鼠時，眼睛始終在看螢幕上的變化，手根據眼睛看到的資訊調整動作。iFay 也是如此：每做一個操作，追蹤器就觀察介面變化，然後模擬操作根據觀察結果決定下一步。

## 它是怎麼工作的

模擬操作的核心工作方式可以用三個關鍵詞概括：**感知、操作、回饋**。

**1. 感知——先看，再動手**

在執行任何操作之前，iFay 先透過第一人稱追蹤器「看」當前介面。它不是去解析網頁的 HTML 程式碼，而是像人一樣**看螢幕上顯示了什麼**：哪裡有按鈕、哪裡有輸入框、哪裡有下拉選單。

**2. 操作——像人一樣互動**

iFay 支持人類在介面上能做的所有操作：
- **點擊**：左鍵點擊按鈕、右鍵開啟選單
- **拖曳**：拖動檔案、調整視窗大小
- **捲動**：上下捲動頁面查看更多內容
- **打字**：在輸入框中輸入文字
- **邊緣手勢**：從螢幕邊緣滑入，呼出側邊欄
- **多指手勢**：雙指縮放、三指切換應用

**3. 回饋——做完之後看結果**

每次操作之後，iFay 都會透過第一人稱追蹤器重新「看」一下介面，確認操作是否成功。如果點擊了一個按鈕但頁面沒有變化，iFay 會意識到「這個按鈕可能沒反應」，然後嘗試其他方式。

這種「操作→觀察→調整」的循環，就是模擬操作與 RPA 腳本的根本區別。RPA 是盲人摸象——按照腳本執行，不管結果如何。iFay 是睜著眼睛幹活——每一步都在看、在想、在調整。

**遇到陌生介面怎麼辦？**

當 iFay 遇到一個從未見過的介面時，它會像人類一樣**探索**：試著點一下這個按鈕看看會發生什麼，把滑鼠懸停在圖示上看看有沒有提示文字，捲動頁面看看下面還有什麼內容。透過這種探索，iFay 逐漸理解介面的結構和功能。

## 與其他模組的關係

| 相關模組 | 關係 | 人體類比 |
|---------|------|---------|
| **第一人稱追蹤器** | 模擬操作的「眼睛」，每次操作前後都透過它感知介面狀態 | 手 ↔ 眼睛（手眼協調） |
| **技能調用** | 同屬動作子系統，但方式不同：模擬操作透過介面操作，技能調用直接呼叫 API | 用手操作 vs 打電話辦事 |
| **自驅行為** | 自驅行為決定「什麼時候動手」，模擬操作負責「怎麼動手」 | 意志決定行動 → 手執行動作 |
| **Ego 模型** | Ego 模型約束模擬操作的行為風格（比如操作速度、謹慎程度） | 性格影響做事方式 |
| **憑證管理** | 模擬操作在登入時需要使用憑證管理提供的帳號密碼 | 手拿鑰匙開門 |

## 場景故事

### 場景一：填寫複雜的保險理賠表

你出了一次小車禍，需要在保險公司的老舊網站上填寫理賠申請。這個網站大概是十年前設計的——表單有 30 多個欄位，分布在 5 個頁面上，還有各種下拉選單和日期選擇器。

你對 iFay 說：「幫我填一下車險理賠，上週三下午在中山路和建設路交叉口的那次。」

iFay 開始工作：
1. 透過第一人稱追蹤器「看」到保險公司網站的登入頁面，用你委託的憑證登入
2. 找到「理賠申請」入口並點擊進入
3. 看到第一頁表單：事故日期、地點、類型……iFay 從你的描述和個人資料堆中提取資訊，逐一填寫
4. 遇到一個「事故照片上傳」按鈕——iFay 點擊它，在彈出的檔案選擇器中找到你拍的現場照片並上傳
5. 翻到第三頁時，發現有個下拉選單的選項和預期不同（網站可能更新過）——iFay 沒有卡住，而是仔細閱讀每個選項的文字，選擇了最匹配的那個
6. 最後一頁是確認頁面，iFay 逐項核對填寫內容，確認無誤後點擊「提交」

整個過程中，iFay 不是在執行一個預先寫好的腳本，而是像一個熟練的助手一樣，**看著螢幕、理解內容、做出判斷**。即使這個網站明天改版了，iFay 依然能完成任務——因為它是「看」著操作的，不是「背」著操作的。

### 場景二：在手機上點外賣，適應 App 更新

你說：「幫我在外送平台上點一份上次吃的那個黃燜雞，送到公司。」

iFay 打開外送 App，但發現介面和上次不一樣了——App 剛更新過，首頁佈局變了，搜尋框的位置也換了。

如果是 RPA 腳本，這時候就會報錯停止。但 iFay 不會：
1. 它透過第一人稱追蹤器掃描新的首頁佈局，發現搜尋圖示移到了右上角
2. 點擊搜尋圖示，輸入「黃燜雞」
3. 在搜尋結果中，透過視覺識別找到你上次點過的那家店（店名和 logo 匹配）
4. 進入店鋪頁面，發現菜單也重新排列了——iFay 捲動瀏覽菜單，找到「黃燜雞米飯」
5. 點擊加入購物車，確認送餐地址是你的公司地址
6. 完成下單

iFay 在整個過程中展現了**自適應探索**能力：它不依賴固定的介面佈局，而是透過視覺感知理解當前介面，靈活調整操作策略。這就是「感知驅動」和「腳本驅動」的本質區別。

## 對開發者

模擬操作屬於 **階段 1（模擬人類操作）** 的核心模組。

- **需求編號**：需求 5（模擬操作）
- **介面規範**：`SimulatedOperation` 介面，包含 `execute()`、`explore()` 和 `getPostActionState()` 三個核心方法
- **支持的操作類型**：`click`、`drag`、`scroll`、`gesture`（邊緣手勢和多指手勢）、`type`
- **關聯模組**：第一人稱追蹤器（`FirstPersonTracer`）提供視覺回饋，憑證管理（`CredentialManager`）提供登入憑證
- **核心設計**：感知驅動而非腳本驅動；每次操作後透過追蹤器感知狀態變化，形成「操作→感知→調整」閉環；支持未知介面的自適應探索
- **合規性測試**：iFACTS L1 驗證各操作類型的執行能力，L2 驗證與第一人稱追蹤器的手眼協調介面
- **iFay Ready 關聯**：Bronze 級認證要求應用支持 iFay 透過模擬操作進行操控
