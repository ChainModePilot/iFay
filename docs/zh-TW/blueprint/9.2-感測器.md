# 9.2 感測器

## 一句話定義

感測器是 iFay 的**神經系統**——它讓 iFay 能感知周圍環境的一切變化，從溫度、位置到心率、光線，就像你身體裡遍布全身的神經末梢。

## 為什麼需要它

如果第一人稱追蹤器是 iFay 的眼睛和耳朵，那感測器就是 iFay 的**整個神經網路**。

想想你的身體：你不只靠眼睛和耳朵感知世界。你的皮膚能感受溫度和觸碰，你的內耳能感知平衡和加速度，你的身體能告訴你餓了、累了、冷了。這些感覺不是來自眼睛或耳朵，而是來自遍布全身的神經系統。

現在想想你的手機和智慧手錶：GPS 知道你在哪裡，加速度計知道你在走路還是跑步，光線感測器知道你在室內還是室外，心率感測器知道你的心跳快不快。這些都是「感測器」——它們是裝置的神經末梢。

iFay 的感測器模組，就是把所有這些裝置上的感測器**統一接入**，讓 iFay 擁有一個完整的神經系統。而且，這個神經系統是可以不斷擴展的——未來出現任何新型感測器，都可以接入進來。

## 它在架構中的位置

```
iFay 四層架構
├── 社交層
├── 互動層          ← 感測器在這裡
│   ├── 感知（Sense）
│   │   ├── 第一人稱追蹤器   ← 向外看，看螢幕
│   │   ├── 👉 感測器        ← 感知環境（溫度、位置、運動……）
│   │   └── 自我感知         ← 向內看，讀懂你
│   └── 動作（Action）
│       ├── 模擬操作
│       ├── 技能調用
│       └── 自驅行為
├── 認知層
│   ├── 思維
│   │   ├── 個人資料堆      ← 感測器資料最終儲存在這裡
│   │   └── ...
│   └── 技能
│       ├── 設備驅動中樞    ← 感測器的硬體介面由它管理
│       └── ...
└── 自我層
```

感測器位於**互動層的感知子系統**中，和第一人稱追蹤器、自我感知並列。但感測器有一個特別之處：它本身只負責**調節靈敏度**（決定什麼時候多收集資料、什麼時候少收集），而實際的硬體連接和資料儲存分別由認知層的**設備驅動中樞**和**個人資料堆**負責。

這就像你的神經系統：神經末梢負責感知，但訊號的傳導通路（驅動中樞）和記憶儲存（資料堆）是由其他系統完成的。

## 它是怎麼工作的

感測器的工作方式可以用三個關鍵詞概括：**橋接、調節、擴展**。

**1. 橋接——連接裝置上的感測器**

你的手機有 GPS、陀螺儀、加速度計、光線感測器、氣壓計……你的智慧手錶有心率感測器、血氧感測器、皮膚溫度感測器……你的智慧家居有溫度感測器、濕度感測器、門窗感測器……

感測器模組就像一個「翻譯官」，把所有這些不同裝置、不同類型的感測器資料，統一翻譯成 iFay 能理解的格式。它基於 **CAP（控制權限協議）** 和 **DTP（資料隧道協議）** 來實現這種橋接。

**2. 調節——動態靈敏度**

你的神經系統有一個很聰明的特性：當你專注工作時，你幾乎感覺不到椅子的觸感；但當有人拍你肩膀時，你立刻就能感覺到。這就是靈敏度的動態調節。

iFay 的感測器也是如此。它不需要時時刻刻以最高精度採集所有感測器的資料——那樣太浪費資源了。比如：
- 你在辦公室安靜工作時，GPS 不需要每秒更新一次位置
- 但當你在開車導航時，GPS 需要高頻更新
- 你睡覺時，心率感測器可以降低取樣頻率
- 但當偵測到心率異常時，立刻提高取樣頻率

感測器模組會根據當前的場景和需求，**自動調節每個感測器的靈敏度**。

**3. 擴展——未來的感測器也能接入**

今天的感測器是 GPS 和心率計，明天可能是腦電波感測器、空氣品質感測器、甚至情緒識別感測器。感測器模組的設計是開放的——當新的感測器類型出現時，只需要透過設備驅動中樞註冊新的驅動，感測器模組就能管理它的靈敏度，個人資料堆就能儲存它的資料。

## 與其他模組的關係

| 相關模組 | 關係 | 人體類比 |
|---------|------|---------|
| **設備驅動中樞** | 感測器的實際硬體介面由設備驅動中樞管理 | 神經末梢 → 神經傳導通路 |
| **個人資料堆** | 感測器採集的資料最終儲存在個人資料堆中 | 感覺訊號 → 記憶儲存 |
| **第一人稱追蹤器** | 同屬感知子系統，但分工不同：追蹤器看螢幕，感測器感知物理環境 | 眼睛 vs 全身神經 |
| **自我感知** | 感測器提供環境資料，自我感知利用這些資料推斷人類原型狀態 | 神經系統提供感覺 → 大腦解讀情緒 |
| **CAP / DTP 協議** | 感測器基於這兩個協議實現資料橋接 | 神經訊號的傳輸協議 |

## 場景故事

### 場景一：智慧健康提醒

下午三點，你已經在電腦前坐了三個小時。iFay 透過你智慧手錶上的感測器資料發現：

- **加速度計**：過去 3 小時幾乎沒有大幅運動（說明你一直坐著）
- **心率感測器**：心率從正常的 72 bpm 升高到 85 bpm（可能是久坐導致的輕微不適）
- **皮膚溫度感測器**：手腕溫度略有升高

感測器模組把這些資料彙總後傳遞給 iFay 的認知層。iFay 綜合判斷後，溫和地提醒你：「你已經坐了 3 個小時了，心率有點偏高，要不要站起來走走？我幫你泡杯茶的時間剛好夠你活動一下。」

在這個過程中，感測器模組並沒有「理解」這些資料意味著什麼——它只是忠實地採集和傳遞。理解和判斷是認知層的工作。但沒有感測器提供的原始資料，認知層就是「巧婦難為無米之炊」。

### 場景二：無人機自主飛行

iFay 被部署在一架無人機上執行航拍任務。這架無人機上有多個感測器：

- **GPS**：提供位置和高度資訊
- **IMU（慣性測量單元）**：提供加速度和角速度，用於姿態控制
- **攝影機**：提供視覺資訊（這部分由第一人稱追蹤器處理）
- **超音波/雷射雷達**：提供障礙物距離資訊
- **氣壓計**：提供精確的高度資料
- **風速感測器**：提供當前風力資訊

感測器模組把所有這些資料流統一接入。在平穩飛行時，IMU 的靈敏度可以適當降低；但當偵測到突然的風速變化時，感測器模組立刻提高 IMU 和氣壓計的靈敏度，讓 iFay 能更精確地控制無人機姿態，避免失控。

## 對開發者

感測器模組屬於 **階段 2（直接接管用戶端）** 的核心模組，依賴 CAP 和 DTP 協議。

- **需求編號**：需求 7（感測器模組）
- **介面規範**：`SensorModule` 介面，包含 `registerSource()`、`adjustSensitivity()`、`getDataStream()` 和 `getActiveStatus()` 四個核心方法
- **關聯協議**：CAP（控制權限協議）用於接管感測器硬體，DTP（資料隧道協議）用於雙向資料傳輸
- **關聯模組**：設備驅動中樞（`DeviceDriverHub`）管理實際硬體介面，個人資料堆（`PersonalDataHeap`）儲存感測器資料
- **合規性測試**：iFACTS L1 驗證靈敏度調節能力，L2 驗證與設備驅動中樞和個人資料堆的介面對接
- **設計要點**：感測器模組僅作為靈敏度調節器，不直接管理硬體介面；支持動態靈敏度調節；新感測器類型透過設備驅動中樞整合
