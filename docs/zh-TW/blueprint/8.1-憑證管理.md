# 8.1 憑證管理

## 一句話定義

憑證管理是 iFay 的**鑰匙串**——它安全地存放著 iFay 代替你使用各種服務所需的「鑰匙」。但這些鑰匙不是你的原始鑰匙，而是安全副本——即使副本丟了，你的原始鑰匙依然安全。

## 為什麼需要它

想想你的日常生活：你有家門鑰匙、車鑰匙、辦公室門禁卡、銀行卡密碼、各種 APP 的登入帳號……這些都是「憑證」——證明「你有權使用某個東西」的憑據。

現在你想讓 iFay 代替你做事——幫你登入電商平台買東西、幫你用公司信箱發郵件、幫你操控智慧家居。iFay 要做這些事，就需要你的「鑰匙」。但你會把家門鑰匙的原件交給一個助手嗎？

更聰明的做法是：配一把**副本鑰匙**給助手。副本能開門，但如果副本丟了，你可以立刻作廢它，而你的原始鑰匙不受影響。這就是憑證管理的核心思路——**副本機制**。

iFay 的憑證管理不只是一個密碼管理器。它管理七種不同類型的憑證，為每種憑證建立安全副本，標記每個憑證的所有者（是你的還是 iFay 自己的），記錄每次使用的稽核日誌，並在偵測到異常時自動撤銷副本。它是 iFay 安全體系的基石。

## 它在架構中的位置

```
iFay 四層架構
├── 社交層          ← 憑證管理在這裡
│   ├── FayID              ← 憑證類型之一，也是 iFay 的身份標識
│   └── 👉 憑證管理         ← iFay 的「鑰匙串」
├── 互動層
│   └── 動作（Action）
│       ├── 模擬操作        ← 模擬操作時需要憑證登入應用
│       ├── 技能調用        ← 呼叫技能時需要憑證認證
│       └── ...
├── 認知層
│   └── 技能（Skill）
│       └── 註冊技能        ← 註冊技能的預授權依賴憑證
└── 自我層
    └── Ego 模型
```

憑證管理位於**社交層**，和 FayID 並列。社交層定義的是 iFay 與外部世界的「關係」——你是誰（FayID）、你有權做什麼（憑證）。憑證管理向下為互動層和認知層的所有需要認證的操作提供憑證支持。

你可以把社交層想像成一個人的「社會身份」——身份證（FayID）證明你是誰，各種證件和鑰匙（憑證）證明你有權做什麼。憑證管理就是幫你管理這些證件和鑰匙的安全鑰匙串。

## 它是怎麼工作的

憑證管理的工作方式可以用四個關鍵詞概括：**分類、複製、標記、守護**。

**1. 分類——七種憑證類型**

iFay 管理的憑證分為七種類型（隨著迭代可能會添加更多）：

![憑證管理](./illustration/credential-management.png)

- **身份標識（FayID）**：iFay 的全域唯一身份，就像你的身份證號碼
- **帳號 / 密碼**：最常見的憑證類型，用於登入各種應用和服務
- **證書**：數位證書，用於加密通訊和身份驗證
- **授權**：第三方授權（如 OAuth），允許 iFay 代表你存取特定服務
- **存取令牌**：臨時的存取憑據，通常有有效期
- **智能合約**：區塊鏈上的智能合約憑證，用於去中心化服務
- **數位代幣（MeriToken）**：對應社會貢獻度的數位代幣，用於 iFay 生態中的價值交換

**2. 複製——副本機制**

當你把憑證委託給 iFay 時，憑證管理不會直接使用你的原始憑證。它會將原始憑證**交換為對應的副本**，iFay 使用副本進行登入和認證。

這個設計的好處是：
- 如果副本被洩露，你可以立刻撤銷它，而原始憑證不受影響
- 副本可以設定更嚴格的權限範圍（比如只允許讀取，不允許修改）
- 副本的使用可以被完整追蹤和稽核

就像你給保姆配了一把只能開大門的鑰匙，而你自己的萬能鑰匙安全地放在保險箱裡。

**3. 標記——誰的鑰匙**

每種憑證都會標明其原始所有者是**人類原型**還是 **iFay 本身**。

這很重要，因為 iFay 不只是使用你的憑證——它也可以擁有自己的憑證。比如：
- 你的電商平台帳號密碼 → 所有者是**人類原型**（你委託給 iFay 使用的）
- iFay 自己註冊的某個 API 存取令牌 → 所有者是 **iFay**（iFay 自主獲得的）
- 透過你的權限換取的某個服務授權 → 來源是**人類原型權限換取**

人類原型權限和 iFay 權限之間沒有包含關係——iFay 可以擁有你沒有的權限，你也可以擁有沒有委託給 iFay 的權限。

**4. 守護——安全防線**

憑證管理不只是儲存和使用憑證，它還是一道安全防線：

- **稽核追蹤**：每次憑證的使用都會被記錄，你可以隨時查看 iFay 用你的憑證做了什麼
- **異常偵測**：如果偵測到憑證副本被異常使用（比如從未知地點登入），系統會立刻撤銷該副本並通知你
- **隱私保護**：當 iFay 被授權查詢隱私資料時，它只會回傳**布林結果**（真或假），而不會暴露實際資料

最後一點特別重要。比如有人聲稱自己的身份證號是 XXXXXX，你讓 iFay 去驗證。iFay 查詢後只會告訴你「是」或「不是」——它不會把那個人的真實身份證號告訴你。這確保了即使 iFay 有權存取敏感資料，資料本身也不會洩露。

## 與其他模組的關係

| 相關模組 | 關係 | 人體類比 |
|---------|------|---------|
| **FayID** | FayID 是七種憑證類型之一，也是 iFay 的核心身份標識 | 身份證是最基礎的證件 |
| **註冊技能** | 註冊技能的預授權依賴憑證管理提供的認證憑證 | 上崗需要出示證件 |
| **模擬操作** | 模擬操作在登入應用時需要使用憑證 | 開門需要鑰匙 |
| **技能調用** | 技能調用在執行任務時需要憑證進行認證 | 辦事需要帶證件 |
| **iFay Profile** | 憑證資訊記錄在 Profile 的權限維度中 | 證件列表是身份證的一部分 |

## 場景故事

### 場景一：委託電商平台帳號——副本機制的安全保障

你想讓 iFay 幫你在電商平台上買日用品。你需要把電商平台帳號委託給 iFay。

委託過程中，憑證管理做了以下事情：
1. 接收你的電商平台帳號密碼（原始憑證）
2. 產生一個對應的**副本憑證**——這個副本可以登入你的電商平台帳號，但權限被限制為「僅限購買日用品類目，單筆不超過 200 元」
3. 將副本憑證安全儲存，標記所有者為「人類原型」
4. 你的原始密碼不會被 iFay 直接持有

一個月後，iFay 幫你買了洗衣精、衛生紙、垃圾袋……一切正常。但某天，憑證管理偵測到一個異常：有人試圖用這個副本憑證購買一台 5000 元的電視——這超出了授權範圍。

憑證管理立刻：
- 拒絕了這次購買請求
- 撤銷了這個副本憑證
- 向你發送通知：「你的電商平台憑證副本偵測到異常使用，已自動撤銷。你的原始帳號密碼未受影響，建議檢查帳號安全。」

你的原始密碼始終安全。即使副本被洩露，損失也被控制在最小範圍內。

### 場景二：隱私查詢——只回傳真或假

你是一家公司的 HR，需要核實一位求職者提供的學歷資訊。你讓 iFay 幫你查詢。

iFay 被授權存取學歷驗證資料庫。它發起查詢：「學歷證書編號 XXXXX 是否有效？」

資料庫回傳了完整的學歷資訊——姓名、學校、專業、畢業時間……但憑證管理的隱私保護機制介入了。iFay 只向你回饋一個結果：**「是，該學歷證書有效。」**

你不會看到求職者的其他隱私資訊，iFay 也不會儲存這些資訊。查詢的目的是驗證真偽，而不是取得資料。憑證管理確保了這條邊界不會被突破——無論是有意還是無意。

這個設計背後的理念是：**iFay 有權查詢，但無權洩露**。它可以幫你確認一個事實，但不會把別人的隱私資料暴露給你。

## 對開發者

憑證管理屬於 **階段 1（模擬人類操作）** 的核心模組，是 iFay 最早需要實作的模組之一。

- **需求編號**：需求 3（憑證管理）
- **介面規範**：`CredentialManager` 介面，包含 `delegate()`（委託憑證/建立副本）、`authenticate()`（使用憑證認證）、`revoke()`（撤銷憑證副本）、`queryPrivacy()`（隱私查詢，僅回傳布林值）和 `getAuditLog()`（稽核日誌）五個核心方法
- **七種憑證類型**：`fay_id`、`account_password`、`certificate`、`authorization`、`access_token`、`smart_contract`、`meri_token`
- **憑證所有者**：`host`（人類原型）或 `ifay`（iFay 本身）
- **副本狀態**：`active`（活躍）、`revoked`（已撤銷）、`expired`（已過期）
- **關聯模組**：註冊技能（`RegisteredSkillManager`）的預授權依賴憑證，模擬操作（`SimulatedOperation`）和技能調用（`InvokeSkillService`）在執行時使用憑證認證
- **FayID 獨立專案**：FayID 作為憑證類型之一，同時也是[獨立開源專案](https://github.com/ChainModePilot/FayID)，詳見 [FayID 文件](https://github.com/ChainModePilot/FayID/wiki)
- **設計要點**：副本機制隔離原始憑證風險；隱私查詢僅回傳布林值；所有操作記錄稽核日誌；異常偵測觸發自動撤銷；人類原型權限與 iFay 權限無包含關係
- **合規性測試**：iFACTS L1 驗證七種憑證類型的管理能力和副本機制，L2 驗證與註冊技能和技能調用的認證介面，L4 驗證隱私查詢的布林回傳約束和異常偵測行為

---

### GMChain 重要說明

MeriToken 作為七種憑證類型之一，其底層依賴 GMChain（貢獻度鏈）。GMChain 和 MeriToken 目前處於概念設計階段，尚未進入實作。在 iFay 的早期階段（階段 1-3），憑證管理模組可以在不實作 GMChain/MeriToken 的情況下獨立運作。開發者在實作憑證管理時，應為 `meri_token` 類型預留介面，但無需實作其完整邏輯。
