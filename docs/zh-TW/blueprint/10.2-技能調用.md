# 10.2 技能調用

## 一句話定義

技能調用是 iFay 的**嘴巴和電話**——如果模擬操作是 iFay 用手操作介面，技能調用就是 iFay 直接「打電話」給服務——不需要操作介面，直接呼叫 API 或觸發任務。

## 為什麼需要它

想像你要翻譯一份文件。你有兩種方式：

**方式一**：打開瀏覽器，進入翻譯網站，把文字貼進去，等待翻譯，複製結果。這就是「模擬操作」的方式——透過介面一步步操作。

**方式二**：直接撥打翻譯公司的電話，說「幫我翻譯這段話」，對方翻譯好直接告訴你結果。不需要打開任何網站，不需要操作任何介面。這就是「技能調用」的方式——**直接打電話給服務**。

為什麼需要第二種方式？因為很多時候，操作介面是多餘的中間環節。你真正想要的不是「打開翻譯網站」，而是「把這段話翻譯成英文」。技能調用讓 iFay 跳過介面，直接完成你的目標。

這就像人類社會中的分工：你不需要親自去麵包店揉麵團（操作介面），你只需要打個電話說「送一個麵包過來」（呼叫技能）。技能調用讓 iFay 能夠高效地利用已經註冊好的各種服務——API、工作流、Bot、Agent、App、微服務——就像一個善於打電話辦事的助手。

## 它在架構中的位置

```
iFay 四層架構
├── 社交層
│   └── 憑證管理           ← 呼叫技能時需要憑證認證
├── 互動層          ← 技能調用在這裡
│   ├── 感知（Sense）
│   │   └── ...
│   └── 動作（Action）
│       ├── 模擬操作        ← 用手操作介面
│       ├── 👉 技能調用     ← 直接「打電話」給服務
│       └── 自驅行為        ← 決定何時行動
├── 認知層
│   └── 技能（Skill）
│       └── 註冊技能        ← 呼叫的前提：技能已註冊
└── 自我層
```

技能調用位於**互動層的動作子系統**中，和模擬操作、自驅行為並列。它是 iFay 與外部世界互動的第二種方式——如果模擬操作是「用手操作」，技能調用就是「用嘴說話」。

技能調用有一個硬性前提：**只能呼叫已註冊的技能**。就像你只能打通訊錄裡有的電話號碼——如果一個技能沒有被註冊（沒有存入通訊錄），iFay 就無法呼叫它。

## 它是怎麼工作的

技能調用的工作方式可以用三個關鍵詞概括：**匹配、呼叫、記錄**。

**1. 匹配——找到對的技能**

當你表達一個意圖時（比如「幫我翻譯這段話」），iFay 需要從已註冊的技能清單中找到最匹配的技能。這個匹配過程考慮：
- 技能的能力描述是否匹配你的意圖
- 技能的預授權是否有效（能不能立刻用）
- 如果有多個匹配的技能，選擇最適合當前場景的那個

**2. 呼叫——執行技能**

找到匹配的技能後，iFay 透過預授權的憑證直接呼叫它。呼叫過程是即時的——因為預授權在註冊時就完成了，不需要再走認證流程。

六種技能類型的呼叫方式各有不同：
- **API**：直接發送 HTTP 請求
- **工作流**：觸發多步驟流程的第一步
- **Bot**：啟動對話
- **Agent**：下達任務指令
- **APP**：啟動應用並執行操作
- **微服務**：呼叫服務端點

**3. 記錄——貢獻追蹤**

每次技能呼叫完成後，結果會被記錄。在 iFay 的完整生態中（階段 5），這些記錄會透過 GMChain 轉化為貢獻度記錄——你的 iFay 使用了某個技能，技能提供者獲得相應的貢獻度。

## 與其他模組的關係

| 相關模組 | 關係 | 人體類比 |
|---------|------|---------|
| **註冊技能** | 技能調用只能呼叫已註冊的技能，註冊是呼叫的前提 | 只能打通訊錄裡的電話 |
| **模擬操作** | 同屬動作子系統，但方式不同：模擬操作透過介面，技能調用直接呼叫 | 用手操作 vs 打電話辦事 |
| **自驅行為** | 自驅行為決定「什麼時候呼叫」，技能調用負責「怎麼呼叫」 | 意志決定行動 → 嘴巴執行 |
| **憑證管理** | 技能調用在執行時需要憑證進行認證 | 打電話需要報上身份 |
| **內部技能** | 內部技能會審核技能調用的輸出，確保符合人類原型風格 | 直覺審核外部建議 |

## 場景故事

### 場景一：翻譯一封日語郵件

你收到一封日語郵件，看不懂。你對 iFay 說：「幫我翻譯一下這封郵件。」

iFay 的技能調用模組開始工作：
1. **匹配**：從已註冊技能中找到「翻譯 API」技能，確認預授權有效
2. **呼叫**：將郵件內容發送給翻譯 API，指定源語言為日語、目標語言為中文
3. **結果**：兩秒後，翻譯結果返回。iFay 把翻譯好的中文呈現給你

整個過程不需要打開任何翻譯網站，不需要複製貼上，不需要等待頁面載入。技能調用讓 iFay 跳過了所有中間步驟，直接完成目標。

### 場景二：多技能協作完成複雜任務

你說：「幫我安排下週三和張總的午餐會議，找一家安靜的日料餐廳，然後發邀請給他。」

這個任務需要多個技能協作：
1. **日曆技能**：查看你和張總下週三的空閒時段，找到共同可用的午餐時間
2. **餐廳推薦技能**：根據「安靜」和「日料」的條件，推薦附近的餐廳
3. **訂位技能**：在選定的餐廳預訂座位
4. **郵件技能**：起草會議邀請郵件，包含時間、地點和餐廳資訊，發送給張總

技能調用模組依序呼叫這四個已註冊技能，每個技能的輸出作為下一個技能的輸入。最終，你只說了一句話，iFay 就完成了查日曆、找餐廳、訂位、發郵件四個步驟。

## 對開發者

技能調用屬於 **階段 2（直接接管用戶端）** 的核心模組。

- **需求編號**：需求 10（技能調用）
- **介面規範**：`InvokeSkillService` 介面，包含 `matchSkill()`（意圖匹配技能）、`invoke()`（呼叫技能）和 `getInvocationLog()`（取得呼叫日誌）三個核心方法
- **呼叫前提**：技能必須已在 `RegisteredSkillManager` 中註冊且預授權有效
- **六種技能類型的呼叫方式**：API（HTTP 請求）、Workflow（觸發流程）、Bot（啟動對話）、Agent（下達指令）、APP（啟動應用）、Microservice（呼叫端點）
- **關聯模組**：註冊技能（`RegisteredSkillManager`）提供可呼叫的技能清單，憑證管理（`CredentialManager`）提供認證支持，內部技能（`InternalSkill`）審核呼叫輸出
- **設計要點**：只能呼叫已註冊技能；意圖匹配支持多技能候選和最優選擇；呼叫結果記錄用於貢獻追蹤
- **合規性測試**：iFACTS L1 驗證意圖匹配和技能呼叫能力，L2 驗證與註冊技能模組的介面對接，L3 驗證「意圖表達→技能呼叫→貢獻記錄」的完整鏈路
